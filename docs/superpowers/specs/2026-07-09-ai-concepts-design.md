## 内容结构

### 一、前言：为什么需要了解这些概念

- 场景引入：很多人直接跟着教程配好了 Claude Code，但遇到问题不知道从哪里排查
- 核心痛点：只会"照抄配置"，不理解背后的模型/API/Agent 是什么
- 本文目标：读完能回答"我在用什么、它怎么工作、出问题怎么定位"

### 二、大模型基础概念

#### 2.1 什么是大模型
- 类比：海量知识压缩进一个"数字大脑"
- 核心特征：参数规模大（数十亿到数千亿）、训练数据广、泛化能力强
- 一句话定义：从海量文本中学习规律，能生成/理解/推理的神经网络

#### 2.2 模型参数是什么
- 类比：大脑的神经元连接数量
- 解释：参数 = 模型学到的"知识权重"，越多越能处理复杂任务
- 常见规模：7B（70亿）、70B（700亿）、400B+（4000亿+）
- 误区澄清：参数多 ≠ 一定更好，还要看训练质量和数据

#### 2.3 训练 vs 推理
- 类比：
  - 训练 = 学骑自行车（摔很多次，调整平衡感）
  - 推理 = 骑车上路（用学好的技能应对新路况）
- 成本差异：训练需要大量算力和数据，推理只需要模型文件+算力运行
- 用户视角：普通开发者只接触推理，训练是大厂/研究机构的事

#### 2.4 模型能力边界
- 能做什么：文本生成、代码编写、问答对话、翻译总结
- 不能做什么：实时联网（除非配工具）、精确计算（会算错）、访问私有数据
- 关键认知：模型是"概率生成器"，不是"真理机器"——会自信地胡说

### 三、API 基础概念

#### 3.1 什么是 API
- 类比：餐厅点餐——菜单是接口，厨房是服务，服务员传话
- 技术定义：应用程序之间的"对话协议"，规定怎么请求、怎么返回
- 为什么需要 API：不用了解模型内部，按格式发请求就能用

#### 3.2 API Key、Base URL
- API Key：你的"身份令牌"，证明你有权限调用
- Base URL：餐厅的"地址"，告诉请求发到哪里
- 类比：API Key = 会员卡，Base URL = 餐厅地址

#### 3.3 请求/响应格式
- 请求：你问什么（prompt）、用什么模型、温度参数等
- 响应：模型回答什么、用了多少 token、是否完成
- 格式：JSON，机器和人都能读的结构化文本

#### 3.4 OpenAI 兼容格式
- 解释：OpenAI 的 API 设计成了行业事实标准
- 兼容格式 = 其他厂商模仿 OpenAI 的接口格式
- 好处：一个客户端（如 Claude Code）能接多家提供商，不用改代码

### 四、Agent 概念

#### 4.1 什么是 AI Agent
- 类比：普通 Chatbot 是"问答机器人"，Agent 是"能动手办事的助手"
- 核心区别：Agent 能调用工具、执行操作、多步骤完成任务

#### 4.2 Agent vs 普通 Chatbot
- Chatbot：你说一句，它答一句，对话结束
- Agent：你说"帮我查天气并订外卖"，它会——查天气 API → 选餐厅 → 下单 → 告诉你结果

#### 4.3 Agent 的工作流
- 感知（Perceive）：接收用户输入和环境信息
- 思考（Think）：规划步骤、选择工具
- 行动（Act）：调用工具、执行操作
- 循环：根据结果调整，直到任务完成

#### 4.4 Tool Use（工具调用）
- 解释：模型不只会说话，还能"伸手"调用外部功能
- 常见工具：文件读写、代码执行、网络搜索、数据库查询
- 关键机制：模型生成"调用指令"，外部程序执行后把结果喂回给模型

### 五、使用中的关键概念

#### 5.1 Token 是什么
- 类比：文字碎片——英文按单词/子词切，中文按字或词切
- 计费单位：API 按 token 数量收费（输入 + 输出分别计）
- 经验值：1 个汉字 ≈ 1-2 token，1 个英文单词 ≈ 1-1.5 token

#### 5.2 上下文窗口（Context Window）
- 类比：模型的"短期记忆力"
- 定义：模型一次能处理的 token 上限（如 4K、128K、200K）
- 重要性：窗口太小，长文档分析不了；窗口大，能一次读整本书
- 注意：上下文窗口 = 输入 + 输出 的总和，不是只看输入

#### 5.3 温度参数（Temperature）
- 类比：创造力的"旋钮"
- 范围：0 ~ 2（常见 0~1）
- 低温度（0~0.3）：保守、确定性强、适合代码/事实问答
- 高温度（0.7~1）：创意多、随机性强、适合写作/头脑风暴

#### 5.4 系统提示词（System Prompt）
- 类比：给模型的"角色设定"或"工作手册"
- 作用：在对话开始前植入指令，定义模型的行为方式
- 示例："你是一个严谨的代码审查员" vs "你是一个幽默的聊天伙伴"
- 用户通常看不到系统提示词，但它深刻影响输出风格

### 六、生态位置关系（一张图讲清）

#### 6.1 全景图

```
┌─────────────────────────────────────────────────────────┐
│                        用户层                            │
│         （你：用 Claude Code、网页 Chat、App）              │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                      应用/工具层                          │
│    Claude Code / ChatGPT / Kimi / 各种 AI 应用           │
│    （负责：界面交互、Agent 编排、Tool 管理）               │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                       API 网关层                          │
│    OpenAI / DeepSeek / 讯飞 / 硅基流动 / OpenRouter       │
│    （负责：认证、计费、路由、格式转换）                     │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                        模型层                             │
│    GPT-4 / Claude / DeepSeek / 星火 / GLM / Qwen         │
│    （负责：理解输入、生成输出、推理计算）                    │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                        算力层                             │
│    NVIDIA GPU / 华为昇腾 / 云服务集群                      │
│    （负责：运行模型的硬件基础设施）                         │
└─────────────────────────────────────────────────────────┘
```

#### 6.2 各层关系说明
- 用户通过工具层调用 API 网关
- API 网关把请求路由到具体模型
- 模型在算力层上运行，返回结果逐层传回
- 一个工具可以同时接多个网关（如 Claude Code 可切换 DeepSeek/讯飞）
- 一个网关也可以聚合多个模型（如 OpenRouter、硅基流动）
