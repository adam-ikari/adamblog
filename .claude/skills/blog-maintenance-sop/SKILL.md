# 博客维护 SOP

> 这份是维护本博客时遵循的标准作业流程，从接到写作/修改需求到发布上线。它的作用是把这几轮维护里反复踩过的坑固化成动作顺序，避免每次都靠临场判断。写作风格见 `blog-writing` skill，这里只管"做什么、按什么顺序做"。

## 一、开工前：先判断任务类型

拿到一个需求，先别动笔，先归到下面四类之一，不同类走不同流程：

| 类型 | 特征 | 流程 |
| --- | --- | --- |
| 新写文章 | 从无到有创建一篇 | 走第二节全流程 |
| 改旧文 | 在已有文章上增删改 | 走第三节 |
| 系列整理 | 串多篇为系列、加索引页 | 走第四节 |
| 纯运维 | 改配置、修构建、SEO 调优 | 走第五节 |

判断不清时，默认按"改旧文"处理——它对现有内容最安全。

## 二、新写文章流程

### 1. 先核事实，不凭记忆写

涉及版本号、价格、API 地址、配置字段、命令这些**会变的事实**，先联网核。核到的写为事实，标注来源或"截至查阅时"；核不到的，以方向性描述呈现并标注"以官方文档为准"，**绝不把没核实的细节写成既成事实**。

- 静态页用 WebFetch；JS 渲染页用 `browser-research` skill。
- 命令、配置代码要能跑通——这是技术博客的信用底线。

### 2. 定主次，不均匀用力

动笔前想清楚：哪几段讲透、哪几段略写。一篇文章每个小节等长，是 AI 痕迹最重的特征。主次本身就是态度。

### 3. 按风格写

严格遵循 `blog-writing` skill 的要求，核心守住四条：
- **有作者视角**，不写成中性百科；观点化在措辞和选材里，不另起段讲观点。
- **去 AI 感**：不凑三段式排比、段落不等长、删套话过渡句、有取舍不贪全。
- **步骤要有说明**，不只甩命令——把命令删掉，读者还能不能理解这步在干嘛。
- **措辞书面但不文言**：用"都需要"，不用"皆须"；口语词（"鸡同鸭讲""虚火重""紧箍咒"）收成现代书面表达。

### 4. 格式自检

- frontmatter 字段齐：title/description/category/tags/recommend/date，系列文章加 series 块。
- 代码块带语言标记，不写裸 ``` ```。
- 提示框 `:::tip`/`:::warning` 语法对、闭合对，用在真正要强调处。
- 内部链接用相对路径不带 .md（cleanUrls 已开）。
- 禁用 ASCII 艺术图，用 Mermaid。

### 5. 构建验证

写完跑 `pnpm build`，确认无语法错误（frontmatter 错、提示框不闭合都会让构建失败）。

### 6. 提交

新文章单独一个 commit，message 用 `feat: ...` 前缀。不要把新文章和改旧文混在一个 commit 里。

## 三、改旧文流程

### 1. 先读全文，别局部盲改

改之前把整篇读一遍，理解原结构再动。局部盲改最易引入矛盾——前后呼应没了、重复内容没删干净。

### 2. 对照差异，避免引入矛盾

改完读一遍改动 diff，自问：
- 改动和原文其他部分有没有冲突？
- 有没有因为删了某段，导致别处引用悬空？
- 重复内容（两篇讲同一主题）是否需要合并而非各改各的？

### 3. 精简优先

删内容比加内容更需要决心。可选操作、次要方法，该删就删，别留着占篇幅误导读者。几轮维护里删过"设置默认用户""常用工具安装"这类，都是可选内容拖累主线。

### 4. 构建验证 + 提交

改旧文单独 commit，message 用 `docs: ...` 或 `refactor: ...`。删内容多时用 `refactor`。

## 四、系列整理流程

### 1. 系列文章用 series 串

多篇相关文章用 frontmatter 的 `series` 块串 prev/next，并在 `series/` 目录维护系列索引页（含 `<SeriesCardList>` 组件）。

### 2. 新增一篇到已有系列

新文章的 series 块要正确指向前后篇，同时把前后篇的 next/prev 同步更新。漏改邻接文章的链接是系列整理最常见的错。

### 3. 避免系列内重叠

系列里两篇文章主题高度重叠时，合并而非并存。判定标准：读者读一篇能不能跳过另一篇——能跳过就该合并。

## 五、纯运维流程

### 1. 改配置/修构建

改 `.vitepress` 配置、依赖、脚本时，改完必须 `pnpm build` 验证。

### 2. SEO 调优

修 meta description、alt 属性、h1 结构这类，改完检查是否引入新的重复（如多篇 description 因 `#` 截断变成相同）。

### 3. 提交

运维改动单独 commit，message 用 `chore: ...` 或 `fix: ...`。

## 六、提交与发布

### 1. 分开提交

按任务类型分 commit，不要一锅烩。一个 commit 只干一件事——这样回滚和追溯都清楚。

### 2. 提交前自检清单

- [ ] 改动符合 blog-writing 风格（有视角、去 AI 感、步骤有说明、措辞书面不文言）
- [ ] 会变的事实已核验，未核的已标注
- [ ] frontmatter 字段完整
- [ ] 代码块带语言标记
- [ ] 提示框语法闭合
- [ ] 内部链接路径正确
- [ ] `pnpm build` 通过

### 3. 推送

`git push` 后，GitHub 上的 Dependabot 警告（依赖漏洞）属正常提示，与本仓库内容无关，不必每次响应——除非是 .vitepress 或 package.json 的实际依赖问题。

## 七、收尾：值得记忆的事

做完一类维护，如果有**非显然的经验**（某类任务踩了新坑、某配置有副作用、某写作偏好被再次确认），存进 memory，不要只留在这篇对话里。memory 跨会话生效，下次维护自动遵循。

判断标准：这件事是不是"下次还会用到、且不是从代码/git 能看出来的"？是，就存；只是本次对话的临时细节，就不存。
