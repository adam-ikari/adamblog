# 博客文章写作

本博客位于仓库根目录，文章放在 `posts/`，用 VitePress + @sugarat/theme 构建。这个 skill 告诉你**这个项目的文章该怎么写**。

## 关于"风格基准"的重要说明

仓库里已有 40+ 篇文章，但**它们不能都当风格标杆**：

- **2023 年及更早的老文章**：代表作者过去的写作水平，**权重最低**，只作格式参考，不作风格标准。
- **2026 年 6 月的「国产模型」系列等 AI 生成文章**：未经作者个人要求生成，**也不代表作者真实意图**，同样只作格式参考。

**真正的风格基准是作者最新的写作意图**——见下方「行文风格」。当旧文章的写法与本 skill 冲突时，以本 skill 为准。作者的需求会演进，旧文章不会自动更新。

## 行文风格（作者最新意图，最高优先级）

### 核心格调：有作者视角，不是中性百科

这是最关键的一点：**文章要有叙事观点，不能写成中性百科条目或 AI 生成的内容清单**。四条要求合一：

1. **观点融在字里行间**——立场不靠"另起一段讲观点"表达，而是从叙述的措辞、选材、轻重、对比里透出来。
   - 反例（事实+观点两段）："麒麟始于 FreeBSD。这说明它不是 Ubuntu 下游。"
   - 正例（观点化在句子里）："麒麟的起点是 FreeBSD，而非某个 Linux 发行版的下游——这个出身决定了它后来走的是另一条路。"
2. **有叙事观点**——作者有判断、有态度，章节推进有论证逻辑，不是平行罗列要点。读者能感到"有人在带着我理解这件事"。
3. **讲透且有主次**——该深挖的讲透，该略的一笔带过。不平铺、不贪全，宁缺毋滥。一篇文章不该每个小节等长，轻重本身就是态度。
4. **客观且观点**——事实必须硬、可核验（这是客观底线，不可妥协）；但选什么事实、怎么排序、用什么词、和谁对比，就是观点。可以下结论，但结论要由前文叙述自然推出，不硬塞。

### 教程与动手做：要有起伏和趣味性，不能太直白

教程、实操、动手做类文章最容易写成"第一步…第二步…第三步"的流水账，干巴巴没意思。这类文章同样要有起伏：

- **要有节奏起伏**——不是匀速平推。遇到坑、卡点、反直觉的地方放慢讲透；顺利的部分快进。读者跟着你走过"以为简单 → 卡住了 → 原来如此 → 豁然开朗"的过程，才有读下去的动力。
- **带一点故事性**——可以交代动机（"为什么要折腾这个"）、小挫折（"结果第一次就报错"）、转折（"后来发现是认证字段写错了"）。不必硬编剧情，但真实的折腾过程本身就有起伏，把它写出来。
- **埋钩与解答**——先抛出一个现象或疑问让读者好奇，再往下解答；而不是一上来就把结论和盘托出。
- **有作者的人格**——吐槽、调侃、恍然大悟都行，让读者觉得是"一个真人在带着我折腾"，不是机器人在念步骤。
- **仍然准确**——趣味性不能牺牲准确性。命令、配置、步骤必须能跑通，这是底线。起伏在叙述，不在事实。

一句话：**教程要写得像一次有波折的实战记录，而不是一份平铺的操作手册。**

### 事实与诚实底线

- 涉及版本号、价格、API 地址、配置字段等**会变的事实，必须联网核验**，不凭记忆写。核到的写为事实并标注来源/时间；核不到的以"方向性描述"呈现并标注"以官方文档为准"，**绝不把未核实的细节写成既成事实**。
- 不编造使用体验。如果没有亲历，体验部分用"公开评测/社区反馈汇总"形式呈现，明确标注来源性质，不伪装亲历。
- 引用配置代码、命令时，确保它们能跑通——这是技术博客的信用。

### 语气与措辞

- 第二人称自然融入（"你"），口语化但不轻浮，像在跟一个懂技术的同行讲话。
- 可以有判断性措辞（"这是 401 报错的高发原因""自主可控是有代价的"），但判断要有前文支撑。
- 不堆砌套话、不写空洞的"随着 XX 的发展"式开头。前言直入主题。
- emoji 只在提示框里用（见格式），正文慎用。

### 去 AI 感，加强人类质感

AI 写的文章有种识别度很高的"工整感"——面面俱到、四平八稳、每个段落差不多长、每个列表都凑成三项。读者一眼就读得出"这是 AI 写的"。要打破这种感觉，关键不是堆词藻，而是**让文章带上真实人类写作的不规则和体温**。

**识别并消除这些 AI 痕迹：**

- **三段式排比 / 凑数的列表**——AI 爱写"首先…其次…最后…"，或把要点硬凑成 3 个、5 个。人类写东西常常是 2 个就够、或一口气列 7 个不讲究。别为了对称硬凑数量。
- **每段等长**——AI 的段落长度惊人地均匀。人类写作长短交错：一句成段、一个长段、再来个短句。主动制造长短起伏。
- **空心过渡句**——"接下来我们来看…""值得注意的是…""总而言之…"这类承上启下的套话，AI 特别爱用。能删就删，直接跳到下一段内容。
- **过度周全**——AI 怕遗漏，什么都要提一嘴，导致没有重点。人类会明确取舍："这个不重要，跳过""这里细节太多，只说关键的"。敢于略过本身就是人类特征。
- **情绪平淡**——AI 的语气永远温和中立。人类有偏好：会吐槽（"这设计挺反人类的"）、会惊讶（"没想到卡在这一步"）、会自嘲（"我在这上面浪费了一下午"）。允许情绪，但别滥情。
- **完美逻辑链**——AI 每一步都严丝合缝。真实的折腾过程是有弯路的：先试了 A 不行、才想到 B。把这种"走过的弯路"写出来，比直线推导真实得多。

**加强人类质感的具体手法：**

- **口语化的连接**——用"话说回来""其实""偏偏""结果"这类口语词衔接，比"此外""另外""然而"自然。
- **短句断行**——偶尔用一个短句单独成段，制造停顿和强调。"就这么简单。" / "但这次没有。"
- **具体而非抽象**——写"我盯着屏幕看了十分钟"比写"经过一番思考"真实；写"翻遍了文档没找到"比写"查阅相关资料后"有体温。具体细节是 AI 最缺的东西。
- **保留不完美**——真实的认知过程有犹豫、有修正（"我一开始以为是 X，后来才发现是 Y"）。别把过程抹平成事后诸葛亮式的直线。
- **有立场敢下判断**——"我个人更推荐 A，虽然 B 也有道理"——这种带个人倾向的结论，AI 通常不敢给（它爱说"各有优劣，看需求"）。适度表态。

**自检：写完一段，问自己——"这读起来像一个真人在跟我聊天，还是一个客服在念稿子？"** 如果像后者，按上面的方法改。目标不是写得花哨，而是写得"有人在"。

## 格式约定（从现有文章提炼，稳定可复用）

### frontmatter

每篇文章必须有 frontmatter，字段如下：

```yaml
---
title: 文章标题
description: 一句话描述，会出现在列表和 SEO
category: 分类名（如 AI工具、前端、信创、系统配置）
tags: [标签1, 标签2, 标签3]
recommend: false   # 一般写 false
date: YYYY-MM-DD
series:             # 仅系列文章才写，单篇省略
  - id: series-xxx
    name: 系列名
    order: N
    prev: /posts/上一篇
    next: /posts/下一篇
---
```

- `title` 与正文一级标题一致。
- `tags` 是数组，用方括号。
- `category` 是自由文本，可新建（如本次「信创」是该分类首篇）。
- 系列文章用 `series` 块串 prev/next，并在 `series/` 目录维护系列索引页（含 `<SeriesCardList>` 组件）。

### 正文结构骨架

多数文章遵循：**前言 → 分章正文 → 对比表/速查表 → 常见问题（可选）→ 总结**。

- 正文一级标题（`#`）与 title 一致，其下用 `##` 分章、`###` 分小节。
- 前言直入主题：交代背景、为什么写、本文解决什么。一两段即可，不灌水。
- 每章开头一两句点明本章讲什么，再展开。
- 总结收束核心要点，常用要点列表或对比表。

### 代码块

- 必须带语言标记：``` ```bash ``` 最常用，另有 ``` ```json ```、``` ```text ```、``` ```powershell ``` 等。不要写裸 ``` ``` ```。
- 命令行用 `bash` 或 `sh`；纯文本输出/示例用 `text`；配置用对应语言。
- 代码块里配关键注释，但别逐行注释到啰嗦。

### 提示框（VitePress admonition）

用 `:::` 语法突出提示、警告、细节：

```markdown
::: tip 提示
这里写提示内容。
:::

::: warning 注意
这里写需要当心的点。
:::
```

- `tip`：有用的小提示、补充建议。
- `warning`：容易踩坑、需当心的点（如认证字段写错会 401）。
- 用在真正需要强调处，不要泛滥——每篇几个即可。

### 表格

对比、速查用表格，比文字列举清晰。表头简洁，单元格不写长句。例如选型对比、配置速查、报错速查都适合表格。

### 内部链接

引用站内其他文章用相对路径：`[配置详解](/posts/macOS-Claude-Code-配置详解)`（不带 `.md`，cleanUrls 已开启）。系列文章之间互相串联。

### 图片

图片放在 `posts/<文章名>/` 目录下，用相对路径引用：`![描述](/posts/文章名/图片.jpg)`。

## 写作流程建议

1. **先核验事实**：涉及会变的事实先联网查，没核到的明确标注。
2. **定主次**：动笔前想清哪些讲透、哪些略写，不要均匀用力。
3. **观点随文**：写完一段回头读，看观点是化在句子里，还是变成了生硬的"总结观点段"——后者要改掉。
4. **格式校对**：frontmatter 字段齐、代码块带语言、提示框语法对、内部链接路径对。
5. **构建验证**：写完跑 `pnpm build` 确认无语法错误（frontmatter 错、提示框不闭合都会让构建失败）。

## 触发提醒

当用户说"写一篇博客文章 / 写个教程 / 整理成文 / 调查报告 / 润色这篇文章"等，先读本 skill 校准风格与格式，再动笔。尤其当内容偏分析/调查类时，"有作者视角、观点融在字里行间、讲透且有主次"是最该守住的底线——这是区分"作者写的"和"AI 随便生成的"的关键。
