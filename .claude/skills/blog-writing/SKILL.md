# 博客文章写作

本博客位于仓库根目录，文章放在 `posts/`，用 VitePress + @sugarat/theme 构建。这个 skill 告诉你**这个项目的文章该怎么写**。

## 关于"风格基准"的重要说明

仓库里已有 40+ 篇文章，但**它们不能都当风格标杆**：

- **2023 年及更早的老文章**：代表作者过去的写作水平，**权重最低**，只作格式参考，不作风格标准。
- **2026 年 6 月的「国产模型」系列等 AI 生成文章**：未经作者个人要求生成，**也不代表作者真实意图**，同样只作格式参考。

**真正的风格基准是作者最新的写作意图**——见下方「行文风格」。当旧文章的写法与本 skill 冲突时，以本 skill 为准。作者的需求会演进，旧文章不会自动更新。

## 行文风格（作者最新意图，最高优先级）

### 核心格调：有作者视角，不是中性百科

这是最关键的一点：**文章要有叙事观点，不能写成中性百科条目或 AI 生成的内容清单**。四条要求合一：

1. **观点融在字里行间**——立场不靠"另起一段讲观点"表达，而是从叙述的措辞、选材、轻重、对比里透出来。
   - 反例（事实+观点两段）："麒麟始于 FreeBSD。这说明它不是 Ubuntu 下游。"
   - 正例（观点化在句子里）："麒麟的起点是 FreeBSD，而非某个 Linux 发行版的下游——这个出身决定了它后来走的是另一条路。"
2. **有叙事观点**——作者有判断、有态度，章节推进有论证逻辑，不是平行罗列要点。读者能感到"有人在带着我理解这件事"。
3. **讲透且有主次**——该深挖的讲透，该略的一笔带过。不平铺、不贪全，宁缺毋滥。一篇文章不该每个小节等长，轻重本身就是态度。
4. **客观且观点**——事实必须硬、可核验（这是客观底线，不可妥协）；但选什么事实、怎么排序、用什么词、和谁对比，就是观点。可以下结论，但结论要由前文叙述自然推出，不硬塞。

### 教程与动手做：要有起伏和趣味性，不能太直白

教程、实操、动手做类文章最容易写成"第一步…第二步…第三步"的流水账，干巴巴没意思。这类文章同样要有起伏：

- **要有节奏起伏**——不是匀速平推。遇到坑、卡点、反直觉的地方放慢讲透；顺利的部分快进。读者跟着你走过"以为简单 → 卡住了 → 原来如此 → 豁然开朗"的过程，才有读下去的动力。
- **带一点故事性**——可以交代动机（"为什么要折腾这个"）、小挫折（"结果第一次就报错"）、转折（"后来发现是认证字段写错了"）。不必硬编剧情，但真实的折腾过程本身就有起伏，把它写出来。
- **埋钩与解答**——先抛出一个现象或疑问让读者好奇，再往下解答；而不是一上来就把结论和盘托出。
- **有作者的人格**——吐槽、调侃、恍然大悟都行，让读者觉得是"一个真人在带着我折腾"，不是机器人在念步骤。
- **仍然准确**——趣味性不能牺牲准确性。命令、配置、步骤必须能跑通，这是底线。起伏在叙述，不在事实。

一句话：**教程要写得像一次有波折的实战记录，而不是一份平铺的操作手册。**

### 步骤要有说明，不能只甩命令（最易犯的毛病）

教程最常见的失败模式：每一步只给"执行这个命令/做这个动作"，命令堆叠得齐齐整整，但**说明缺位**——读者照着能做下来，却不知道为什么这么做、这步在整体里起什么作用、关键参数什么意思、出问题怎么判断。这就是"流水账教程"：步骤齐全，价值为零。

每一步至少要回答其中几条，不能只甩命令就完事：

- **为什么这一步必要**——它在整体流程里解决什么、不做会怎样。比如配 `ANTHROPIC_DEFAULT_HAIKU_MODEL`，不只说"加上这行"，要说清不映射会导致后台小任务找不到模型而报错。
- **关键参数/选项什么意思**——命令里非显然的部分要解释。`curl` 的 `-H "Authorization: Bearer ..."` 为什么是 Bearer 而非别的，比命令本身重要。
- **预期结果长什么样**——执行完该看到什么输出，让读者能判断"我跑对了没有"。只给命令不给预期，读者跑出报错都不知道是哪步错了。
- **踩坑点**——这一步容易在哪卡住、错了怎么排查。把"我在这儿栽过"写出来，比十句顺利的步骤有用。

反面教材（避免）：`## 第一步：确认环境` 下只有一段 `node --version`，再加一句"没装好回前面教程补齐"——读者知道了怎么查版本，却没学到任何判断标准或排错思路。

正面做法：同样这一步，补上"版本低于 18 会怎样、怎么从报错判断是版本问题、nvm 切换后为何要重开终端"这些说明。命令是骨架，说明是肉，光有骨架立不起来。

判断标准：**把每一步的命令全删掉，光读说明文字，读者还能不能理解这一步在干嘛、为什么。** 能，就是好教程；只剩一堆空标题，就是流水账。

### 事实与诚实底线

- 涉及版本号、价格、API 地址、配置字段等**会变的事实，必须联网核验**，不凭记忆写。核到的写为事实并标注来源/时间；核不到的以"方向性描述"呈现并标注"以官方文档为准"，**绝不把未核实的细节写成既成事实**。
- 不编造使用体验。如果没有亲历，体验部分用"公开评测/社区反馈汇总"形式呈现，明确标注来源性质，不伪装亲历。
- 引用配置代码、命令时，确保它们能跑通——这是技术博客的信用。

### 语气与措辞

- 第二人称自然融入（"你"），口语化但不轻浮，像在跟一个懂技术的同行讲话。
- 可以有判断性措辞（"这是 401 报错的高发原因""自主可控是有代价的"），但判断要有前文支撑。
- 不堆砌套话、不写空洞的"随着 XX 的发展"式开头。前言直入主题。
- emoji 只在提示框里用（见格式），正文慎用。

### 文风的度：书面但不文言

文风要在"通俗口语"和"书面专业"之间找点，**别矫枉过正**。一个常见失误：从通俗往书面调时，过度堆用文言虚词——"皆须""亦""之""于"连用，句子变成"两边皆须维护、皆须投入"这种半文半白，比通俗更难读。

- 书面 ≠ 文言。用"都需要维护和投入"，不用"皆须维护、皆须投入"。
- 文言虚词（皆、亦、之、于、其）偶用可增凝练感，连用即成做作。一句里最多一个。
- 判断句保持现代汉语骨架（"是…的""要…而非…"），别硬扭成"者…也"式。
- 自检：读出声来，像现代成年人写的书面文字，而非仿古作文。若某句读着别扭、像在"装"，就改回自然的现代书面表达。

这条是在 KylinOS 报告文风重写中踩过的坑：为去通俗而过度文言化，反而生硬。记于此，戒之。

### 去 AI 感，加强人类质感

AI 写的文章有种识别度很高的"工整感"——面面俱到、四平八稳、每个段落差不多长、每个列表都凑成三项。读者一眼就读得出"这是 AI 写的"。要打破这种感觉，关键不是堆词藻，而是**让文章带上真实人类写作的不规则和体温**。

**识别并消除这些 AI 痕迹：**

- **三段式排比 / 凑数的列表**——AI 爱写"首先…其次…最后…"，或把要点硬凑成 3 个、5 个。人类写东西常常是 2 个就够、或一口气列 7 个不讲究。别为了对称硬凑数量。
- **每段等长**——AI 的段落长度惊人地均匀。人类写作长短交错：一句成段、一个长段、再来个短句。主动制造长短起伏。
- **空心过渡句**——"接下来我们来看…""值得注意的是…""总而言之…"这类承上启下的套话，AI 特别爱用。能删就删，直接跳到下一段内容。
- **过度周全**——AI 怕遗漏，什么都要提一嘴，导致没有重点。人类会明确取舍："这个不重要，跳过""这里细节太多，只说关键的"。敢于略过本身就是人类特征。
- **情绪平淡**——AI 的语气永远温和中立。人类有偏好：会吐槽（"这设计挺反人类的"）、会惊讶（"没想到卡在这一步"）、会自嘲（"我在这上面浪费了一下午"）。允许情绪，但别滥情。
- **完美逻辑链**——AI 每一步都严丝合缝。真实的折腾过程是有弯路的：先试了 A 不行、才想到 B。把这种"走过的弯路"写出来，比直线推导真实得多。

**加强人类质感的具体手法：**

- **口语化的连接**——用"话说回来""其实""偏偏""结果"这类口语词衔接，比"此外""另外""然而"自然。
- **短句断行**——偶尔用一个短句单独成段，制造停顿和强调。"就这么简单。" / "但这次没有。"
- **具体而非抽象**——写"我盯着屏幕看了十分钟"比写"经过一番思考"真实；写"翻遍了文档没找到"比写"查阅相关资料后"有体温。具体细节是 AI 最缺的东西。
- **保留不完美**——真实的认知过程有犹豫、有修正（"我一开始以为是 X，后来才发现是 Y"）。别把过程抹平成事后诸葛亮式的直线。
- **有立场敢下判断**——"我个人更推荐 A，虽然 B 也有道理"——这种带个人倾向的结论，AI 通常不敢给（它爱说"各有优劣，看需求"）。适度表态。

**自检：写完一段，问自己——"这读起来像一个真人在跟我聊天，还是一个客服在念稿子？"** 如果像后者，按上面的方法改。目标不是写得花哨，而是写得"有人在"。

### AI 词汇与句式黑名单（可对照删除）

下面这些是 AI 写作的高频痕迹，源自 `humanizer-zh` skill（基于维基百科"AI 写作特征"）。写完拿这份清单扫一遍，命中就改：

**AI 词汇黑名单**（技术博客里尤甚，能删就删或换朴素说法）：
此外、值得注意的是、至关重要、深入探讨、强调、彰显、不断演变的格局、关键作用、织锦、充满活力、自然之美、致力于、开创性的、令人叹为观止、凸显、为……奠定基础、标志着……的转变。

**口语/生造词禁用**（严谨调查报告里不该出现，写作中禁用）：
- **根子**（如"根子在采购""根子却在"）——生硬口语，改"根源""关键在于"或重写句子。
- **干活卡手/卡手**——方言口语，改"实际使用受阻""落地不畅"。
- **喂熟**（如"生态需时间喂熟"）——比喻不当，改"培育""成熟"。
- **一摊/一滩**（如"台式机那一摊"）——口语，改"领域""方面"。
- **那一摊/这摊**——同上。
- **"最X的一块/一层/一环"**（如"国密替代最实的一块""信创里进展最实的一层"）——AI 式工整概括腔，用"最…的之一"笼统收束、空泛。改为具体陈述：说清是哪一块、实到什么程度、有什么证据，不要用"最X的一块"这种偷懒收束。
- **成色**（如"自主成色""国产成色""成色最硬""成色排序"）——AI 滥用词，把金属鉴定术语套到一切"程度/水平"上，一篇里反复出现尤滥。改"自主程度""国产化程度""含金量"等视语境而定，或在冗余处直接删。
- **核验声明预告**（如"事实尽量核到来源，截至撰写时的公开信息为准，核不到的明确标注，不装懂""事实尽量核至来源，核不到的明说"）——典型 AI 报告腔。核验是做事，不是用来在前言预告的；真核了，文末参考资料和脚注自然会体现，不用提前声明。这类话每篇来一遍是机械重复。直接删掉，前言收在内容本身。
- **形容词叠用**（如"更难、更慢""更高、更快"这种对仗式叠词）——人类写作不会刻意把形容词凑成对仗排列。要么只说一个最关键的，要么用具体描述代替（"更难"→具体说难在哪，"更慢"→给出对比或数据）。判断标准：两个同类形容词用顿号连排，多半是 AI 在凑节奏。
- **破折号过度**——AI 比人类更爱用破折号（——）装"有力"和补充。中文里多用逗号、分句或括号代替。一篇文章里破折号别超过三五个，连续段落都靠破折号转折就是过度。
- 判断标准：读起来像方言口语或生造比喻的，换现代书面表达。严谨文章里不用方言、不用过于随意的比喻，也不用"最X的一块"这类空泛概括，更不在前言预告"我会核验"。

**句式黑名单**：
- **夸大意义**——动辄"标志着""见证了""为……奠定基础"。普通一件事不必拔高到时代意义。
- **-ing / "……着"肤浅分析**——句尾加"彰显了……""反映了……""为……做出贡献"装深度。删掉，陈述事实即可。
- **否定式排比**——"不仅是 X，更是 Y""这不是 A，而是 B"。偶尔用行，AI 是滥用。
- **虚假范围**——"从 X 到 Y"，但 X、Y 不在同一有意义的尺度上。
- **模糊归因**——"专家认为""行业报告显示"却不给具体来源。要么给具体出处，要么别引。
- **通用积极结论**——"未来前景光明""迈出重要一步"。结尾别灌鸡汤，落到具体。
- **内联粗体标题列表**——"- **性能：** 性能得到增强"这种。改成自然句子。
- **破折号过度**——AI 比人类更爱用破折号装"有力"。中文里多用逗号或分句代替。

**填充词替换**："为了实现这一目标"→"为此"；"在这个时间点"→"现在"；"值得注意的是数据显示"→"数据显示"；"由于……的事实"→"因为……"。能短则短。

## 格式约定（从现有文章提炼，稳定可复用）

### frontmatter

每篇文章必须有 frontmatter，字段如下：

```yaml
---
title: 文章标题
description: 一句话描述，会出现在列表和 SEO
category: 分类名（如 AI工具、前端、信创、系统配置）
tags: [标签1, 标签2, 标签3]
recommend: false   # 一般写 false
date: YYYY-MM-DD
series:             # 仅系列文章才写，单篇省略
  - id: series-xxx
    name: 系列名
    order: N
    prev: /posts/上一篇
    next: /posts/下一篇
---
```

- `title` 与正文一级标题一致。
- `tags` 是数组，用方括号。
- `category` 是自由文本，可新建（如本次「信创」是该分类首篇）。
- 系列文章用 `series` 块串 prev/next，并在 `series/` 目录维护系列索引页（含 `<SeriesCardList>` 组件）。

### 正文结构骨架

多数文章遵循：**前言 → 分章正文 → 对比表/速查表 → 常见问题（可选）→ 总结**。

- 正文一级标题（`#`）与 title 一致，其下用 `##` 分章、`###` 分小节。
- 前言直入主题：交代背景、为什么写、本文解决什么。一两段即可，不灌水。
- 每章开头一两句点明本章讲什么，再展开。
- 总结收束核心要点，常用要点列表或对比表。

### 代码块

- 必须带语言标记：``` ```bash ``` 最常用，另有 ``` ```json ```、``` ```text ```、``` ```powershell ``` 等。不要写裸 ``` ``` ```。
- 命令行用 `bash` 或 `sh`；纯文本输出/示例用 `text`；配置用对应语言。
- 代码块里配关键注释，但别逐行注释到啰嗦。

### C++ 代码示例规范

博客里的 C++ 代码示例统一遵循两条：

- **用 C++17 及以后的语法**。优先用 `std::optional`、结构化绑定、`if constexpr`、`[[nodiscard]]`、`std::variant`、`std::string_view`、范围 `for` 等；避免老式 C++ 写法（裸指针 new/delete、C 风格字符串、宏常量等）。体现现代 C++ 的惯用法。
- **风格偏 Java 的 OOP**。类边界清晰、接口用抽象类表达、方法命名完整、依赖通过构造注入。不写模板元编程、不过度用 STL 函数式风格。原因：博客示例重在讲设计思想（如可测试性、依赖注入），接近 Java 的 OOP 写法读者最易对照，模板魔法反而喧宾夺主。

示例骨架：接口用纯虚抽象类，实现类继承它，依赖在构造函数注入，用 `std::unique_ptr`/`std::shared_ptr` 管理所有权，方法加 `[[nodiscard]]`/`const` 标注。

### 提示框（VitePress admonition）

用 `:::` 语法突出提示、警告、细节：

```markdown
::: tip 提示
这里写提示内容。
:::

::: warning 注意
这里写需要当心的点。
:::
```

- `tip`：有用的小提示、补充建议。
- `warning`：容易踩坑、需当心的点（如认证字段写错会 401）。
- 用在真正需要强调处，不要泛滥——每篇几个即可。

### 表格

对比、速查用表格，比文字列举清晰。表头简洁，单元格不写长句。例如选型对比、配置速查、报错速查都适合表格。

### 内部链接

引用站内其他文章用相对路径：`[配置详解](/posts/macOS-Claude-Code-配置详解)`（不带 `.md`，cleanUrls 已开启）。系列文章之间互相串联。

### 图片

图片放在 `posts/<文章名>/` 目录下，用相对路径引用：`![描述](/posts/文章名/图片.jpg)`。

### 图表（Mermaid）

**禁止使用 ASCII 艺术图**（用 `┌┐└┘├┤│─` 等字符画的框图、流程图）。ASCII 图在不同终端/字体下显示错乱，且无法搜索、无法缩放、不利于无障碍阅读。

**替代方案：Mermaid 图表**。博客已集成 `vitepress-plugin-mermaid`，支持在 Markdown 中直接写 Mermaid 语法，构建时渲染为 SVG。

**常用场景与语法**：

- **流程图**（顺序执行、循环判断）：
  ````markdown
  ```mermaid
  flowchart TD
      A[开始] --> B{判断}
      B -->|是| C[执行 A]
      B -->|否| D[执行 B]
      C --> E[结束]
      D --> E
  ```
  ````

- **架构图/层级图**：
  ````markdown
  ```mermaid
  flowchart TD
      A["应用层"] --> B["框架层"]
      B --> C["原生层"]
      C --> D["硬件层"]
  ```
  ````

- **决策树**（多分支选择）：
  ````markdown
  ```mermaid
  flowchart TD
      A["有稳定代理？"] -->|否| B["选国内直连"]
      A -->|是| C["考虑 OpenRouter"]
  ```
  ````

**规则**：
- 所有含 `┌┐└┘├┤┬┴┼│─` 字符的代码块都必须替换为 Mermaid。
- Mermaid 代码块用 `mermaid` 语言标记。
- 节点文字用中文，保持简洁（每节点不超过 15 字）。
- 复杂架构可用 `subgraph` 分组，但避免过度复杂（单图节点不超过 15 个）。
- 目录树（`tree` 命令输出）保留原样——它用 `├──` `└──` 是标准树形表示，不属于需要替换的 ASCII 艺术图。

## 写作流程建议

### 先判断任务类型

拿到需求先归到下面四类，不同类走不同流程。判断不清时默认按"改旧文"处理——它对现有内容最安全。

| 类型 | 特征 | 流程 |
| --- | --- | --- |
| 新写文章 | 从无到有创建一篇 | 走"新写文章"流程 |
| 改旧文 | 在已有文章上增删改 | 走"改旧文"流程 |
| 系列整理 | 串多篇为系列、加索引页 | 走"系列整理"流程 |
| 纯运维 | 改配置、修构建、SEO 调优 | 改完必跑 `pnpm build`，单独提交 |

### 新写文章

1. **先核验事实**：涉及会变的事实先联网查，没核到的明确标注。
2. **定主次**：动笔前想清哪些讲透、哪些略写，不要均匀用力。
3. **观点随文**：写完一段回头读，看观点是化在句子里，还是变成了生硬的"总结观点段"——后者要改掉。
4. **格式校对**：frontmatter 字段齐、代码块带语言、提示框语法对、内部链接路径对。
5. **构建验证**：写完跑 `pnpm build` 确认无语法错误（frontmatter 错、提示框不闭合都会让构建失败）。

### 改旧文

1. **先读全文，别局部盲改**：改之前把整篇读一遍，理解原结构再动。局部盲改最易引入矛盾——前后呼应没了、重复内容没删干净。
2. **对照差异，避免矛盾**：改完读一遍 diff，自问改动和原文其他部分有没有冲突、有没有因删某段导致别处引用悬空、重复内容是否该合并而非各改各的。
3. **精简优先**：删内容比加内容更需要决心。可选操作、次要方法，该删就删，别留着占篇幅误导读者。

### 系列整理

1. 多篇相关文章用 frontmatter 的 `series` 块串 prev/next，并在 `series/` 目录维护索引页。
2. 新增一篇到已有系列时，新文章的 series 块要正确指向前后篇，**同时同步更新前后篇的 next/prev**——漏改邻接文章是系列整理最常见的错。
3. 系列内两篇主题高度重叠时合并而非并存。判定标准：读者读一篇能不能跳过另一篇——能跳过就该合并。

### 提交与发布

1. **分开提交**：按任务类型分 commit，一个 commit 只干一件事，便于回滚和追溯。新文章用 `feat:`，改旧文用 `docs:`/`refactor:`，运维用 `chore:`/`fix:`。不要把新文章和改旧文混在一个 commit。
2. **提交前自检**：风格符合本 skill、会变的事实已核验、frontmatter 完整、代码块带语言、提示框闭合、内部链接正确、`pnpm build` 通过。
3. 推送后 GitHub 上的 Dependabot 依赖漏洞警告，若与本仓库实际依赖无关可忽略；只有涉及 `.vitepress` 或 `package.json` 的才需响应。

## 触发提醒

当用户说"写一篇博客文章 / 写个教程 / 整理成文 / 调查报告 / 润色这篇文章"等，先读本 skill 校准风格与格式，再动笔。尤其当内容偏分析/调查类时，"有作者视角、观点融在字里行间、讲透且有主次"是最该守住的底线——这是区分"作者写的"和"AI 随便生成的"的关键。
