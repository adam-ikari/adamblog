# 【文档翻译】编写可测试代码 · 缺陷二 · 挖掘合作者

> 译自 misko hevery 的 [Guide: Writing Testable Code](https://github.com/mhevery/guide-to-testable-code) 中的 *Flaw #2: Digging into Collaborators*。为贴合 C++ 语境，代码示例由原文的 Java 改写而来，论述忠于原文。

避免"holder"、"context"和"厨房水槽"对象（这类对象收罗了各种其他对象，是一袋子合作者的大杂烩）。把你需要的具体对象作为参数传进来，而不是传一个装着它的 holder。避免从一个对象里伸手去够另一个对象，如此反复。不要靠遍历对象图来寻找目标对象（这会让我们不必要地和所有中间对象亲密接触）。调用链不必很深：如果一条链里你数出的句点多于一个，你正看着的就是这个缺陷的例子。*直接注入你需要的对象，而不是到处溜达去找它们。*

> #### 危险信号
>
> - 对象被传进来却从不直接使用（只用来获取其他对象）
> - 迪米特法则违反：方法调用链遍历对象图、句点多于一个（`.`）
> - 可疑的名字：context、environment、principal、container 或 manager

![挖掘合作者 vs 直接注入：迪米特法则的两面](/posts/【文档翻译】编写可测试代码-缺陷二-挖掘合作者/digging-into-collaborators.svg)

## 为什么这是个缺陷

*欺骗性的 API*

如果你的 API 宣称它只需要一个作为字符串的信用卡号，但代码却偷偷溜出去翻找 `CardProcessor` 或 `PaymentGateway`，真正的依赖就不清晰。把你真正需要的合作者声明在你的 API 里（方法签名，或对象的构造函数）。

*让代码变得脆弱*

想象一下当需要改动时，你有一堆用来翻找、获取所需对象的"中间人"。它们当中许多都得被改，才能 accommodate 新的交互。相反，试着在需要的地方拿到你需要的最具体的对象。（在这个过程中你可能会发现某个类需要被拆开，因为它有不止一项职责。别怕；追求[单一职责原则](https://web.archive.org/web/20200502184502/http://en.wikipedia.org/wiki/Separation_of_concerns)。）另外，当到处翻找所需对象时，你和中间对象变得亲密，它们也就成了编译所需的依赖。

*让你的代码臃肿、把真正发生的事搞复杂*

有了这一堆翻找目标的中间步骤，你的代码更让人困惑，也比它本该有的更长。

*难以测试*

如果你要测一个接收 context 对象的方法，在执行该方法时，很难猜出它从 context 里拽出了什么、什么又是它不在意的。这通常意味着我们传一个空的 context 进去，得到一些空指针，然后在 context 里设点状态再试一遍。如此重复，直到所有空指针都消失。另见 *Breaking the Law of Demeter is Like Looking for a Needle in the Haystack*。

## 识别这个缺陷

这也被称为"火车相撞"（Train Wreck）或"迪米特法则"违反（见维基百科 [Law of Demeter](https://web.archive.org/web/20200502184502/http://en.wikipedia.org/wiki/Law_of_Demeter)）。

- 症状：测试里不得不创建返回 mock 的 mock
- 症状：读到一个名叫"context"的对象
- 症状：在方法都是 getter 的调用链里看到多于一个句点"."
- 症状：因复杂的 fixture 搭建而难写测试

*违反示例：*

```cpp
// 改写自原文 Java 示例
user_manager_->getUser(123).getProfile().isAdmin();
// 这糟糕透顶（你只想知道用户是不是管理员）
```

```cpp
// 改写自原文 Java 示例
context->get_common_data_store()->find(1234); // 这不好
```

## 修复这个缺陷

与其去找东西，不如在构造函数或方法参数里直接要你需要的对象。通过在构造函数里声明依赖，你把"找对象"的职责移给了创建这个类的工厂，通常是一个工厂或 GUICE。

- 只和你直接的朋友说话。
- 注入（传进来）你真正需要的那个更具体的对象。
- 把对象定位与配置的职责留给调用者，即工厂或 GUICE。

## 具体代码示例：改造前后

从根本上说，"挖掘合作者"指的是任何时候你没有手头真正需要的对象、得四处翻找去拿到它。每当代码开始到处问别的对象要东西，这就是一个信号：它将很难测。与其问别的对象去拿你的合作者，不如把合作者作为依赖声明在方法或构造函数的参数里。（别去找东西！去要东西！）

### 问题：服务对象在值对象里挖掘

```cpp
// 改造前：一个服务对象在和值对象（User 与 amount）打交道。

class SalesTaxCalculator {
public:
    explicit SalesTaxCalculator(TaxTable& tax_table)
        : tax_table_(tax_table) {}

    [[nodiscard]] float computeSalesTax(const User& user,
                                        const Invoice& invoice) const {
        // 注意 "user" 从未被直接使用
        const Address& address = user.getAddress();
        float amount = invoice.getSubTotal();
        return amount * tax_table_.getTaxRate(address);
    }

private:
    TaxTable& tax_table_;
};
```

```cpp
// 改造后：它只请求与之合作的那些具体对象。

class SalesTaxCalculator {
public:
    explicit SalesTaxCalculator(TaxTable& tax_table)
        : tax_table_(tax_table) {}

    // 注意我们不再用 User，也不再往 Address 里挖。
    // （注：现实中我们会用 Money、BigDecimal 之类。）
    [[nodiscard]] float computeSalesTax(const Address& address,
                                        float amount) const {
        return amount * tax_table_.getTaxRate(address);
    }

private:
    TaxTable& tax_table_;
};
```

```cpp
// 改造前：测试通过搭建对象图、测你关心的那个小行为
// 所需的大量工作，暴露了问题。

class SalesTaxCalculatorTest {
public:
    void testWithTooMuchWiring() {
        TaxTable tax_table;
        SalesTaxCalculator calc{tax_table};
        // 把所有需要的对象拼起来，工作量好大
        Address address{"1600 Amphitheatre Parkway..."};
        User user{address};
        Invoice invoice{1, ProductX{95.00}};
        // ...
        assert(almostEquals(calc.computeSalesTax(user, invoice), 0.09f, 0.05f));
    }
};
```

```cpp
// 改造后：新 API 把它需要的合作者说得更清楚了。

class SalesTaxCalculatorTest {
public:
    void testWithClearApi() {
        TaxTable tax_table;
        SalesTaxCalculator calc{tax_table};
        // 只拼起需要的那些对象
        Address address{"1600 Amphitheatre Parkway..."};
        // ...
        assert(almostEquals(calc.computeSalesTax(address, 95.00f), 0.09f, 0.05f));
    }
};
```

这个例子把对象查找和计算混在了一起。核心职责是把一个金额乘以税率。

- 缺陷：要测这个类，你得实例化一个 `User` 和一个 `Invoice`，并填进去一个 `Zip` 和一个金额。这是测试的额外负担。
- 缺陷：对于方法的调用者，不清楚所需的全部其实只是一个 `Address` 和一个 `Invoice`。（API 在骗你。）
- 缺陷：从代码复用角度看，如果你想在另一个项目用这个类，还得连带提供 `Invoice`、`User` 这类无关类的源码。（而它们又会拽进更多依赖。）

解决办法是通过方法签名声明交互所需的那些具体对象，仅此而已。

### 问题：服务对象直接违反迪米特法则

```cpp
// 改造前：一个违反迪米特法则的服务对象。

class LoginPage {
public:
    LoginPage(RPCClient& client, HttpRequest& request)
        : client_(client), request_(request) {}

    bool login() {
        std::string cookie = request_.getCookie();
        return client_.getAuthenticator().authenticate(cookie);
    }

private:
    RPCClient& client_;
    HttpRequest& request_;
};
```

```cpp
// 改造后：我们需要的具体对象被直接传了进来。

class LoginPage {
public:
    LoginPage(std::string cookie, Authenticator& authenticator)
        : cookie_(std::move(cookie)), authenticator_(authenticator) {}

    bool login() {
        return authenticator_.authenticate(cookie_);
    }

private:
    std::string cookie_;
    Authenticator& authenticator_;
};
```

```cpp
// 改造前：铺天盖地又复杂的 fake/mock 用法，
// 是设计脆弱的信号。

class LoginPageTest {
public:
    void testTooComplicatedThanItNeedsToBe() {
        auto authenticator = std::make_unique<FakeAuthenticator>();
        // 这个 fake 替身（client）又返回另一个替身，
        // 很糟。
        auto client = std::make_unique<FakeRPCClient>();
        client->setAuthenticator(authenticator.get());
        auto request = std::make_unique<FakeHttpRequest>();
        request->setCookies({{"g", "xyz123"}});
        LoginPage page{*client, *request};
        // ...
        assert(page.login());
    }
};
```

```cpp
// 改造后：现在耦合更松，更可维护、更灵活、也更可测。

class LoginPageTest {
public:
    void testMuchEasier() {
        std::string cookie = "xyz123";
        auto authenticator = std::make_unique<FakeAuthenticator>();
        LoginPage page{cookie, *authenticator};
        // ...
        assert(page.login());
    }
};
```

最常见的迪米特法则违反会有许多链式调用，然而这个例子表明，仅用一条链也能违反。从 `RPCClient` 取 `Authenticator` 就是一种违反，因为 `RPCClient` 在别处没被使用，只用来取 `Authenticator`。

- 缺陷：没人真的在这个类里关心 `RPCClient`。为什么要传它进来？
- 缺陷：没人真的在这个类里关心 `HttpRequest`。为什么要传它进来？
- 缺陷：cookie 才是我们需要的，却得钻进 request 去拿它。对测试来说，实例化一个 `HttpRequest` 并非易事。
- 缺陷：`Authenticator` 才是真正关心的对象，可我们得钻进 `RPCClient` 去拿它。

为了测试原来那段坏代码，我们不得不 mock 掉 `RPCClient` 和 `HttpRequest`。而且测试对实现极其亲密，因为我们得把对象图的遍历都 mock 掉。修好后的代码里，我们不必 mock 任何图遍历。这更容易，也让代码更不脆弱。（即便我们在"改造后"那版里选择 mock `Authenticator`，也更容易，并产生一个更松耦合的设计。）

### 问题：违反迪米特法则、不当地造出一个服务定位器

```cpp
// 改造前：Database 有了单一职责的身份危机。

class UpdateBug {
public:
    explicit UpdateBug(Database& db) : db_(db) {}

    void execute(const Bug& bug) {
        // 到处挖掘，违反迪米特法则
        db_.getLock().acquire();
        try {
            db_.save(bug);
        } catch (...) {
            db_.getLock().release();
            throw;
        }
        db_.getLock().release();
    }

private:
    Database& db_;
};
```

```cpp
// 改造后：修订后的 Database 有了单一职责。

class UpdateBug {
public:
    UpdateBug(Database& db, Lock& lock)
        : db_(db), lock_(lock) {}

    void execute(const Bug& bug) {
        // db 不再有 getLock 方法
        lock_.acquire();
        try {
            db_.save(bug);
        } catch (...) {
            lock_.release();
            throw;
        }
        lock_.release();
    }

private:
    Database& db_;
    Lock& lock_;
};
// 注：在 Database 里，getLock() 方法被移除了
```

```cpp
// 改造前：连正常路径都因为要这么多 mock 对象而复杂，
// 尤其是返回 mock 的 mock（很糟）。

class UpdateBugTest {
public:
    void testThisIsRidiculousHappyPath() {
        Bug bug{"description"};
        // 这既违反迪米特法则，又滥用 mock，
        // 而这些 mock 本不全是必需的。
        auto db = std::make_unique<FakeDatabase>();
        auto lock = std::make_unique<FakeLock>();
        // 哎呀，这个 fake（db）返回另一个 fake。
        db->setLock(lock.get());
        EXPECT_CALL(*lock, acquire());
        EXPECT_CALL(*db, save(bug));
        EXPECT_CALL(*lock, release());
        // 终于把 mock 设好了！
        UpdateBug update_bug{*db, *lock};
        update_bug.execute(bug);
        // 验证它如预期发生
    }
};
```

```cpp
// 改造后：两种改进方案：基于状态的测试
// 和基于行为的（Mockist）测试。

// 第一种方案：基于状态的测试。
class UpdateBugStateBasedTest {
public:
    void testThisIsMoreElegantStateBased() {
        Bug bug{"description"};
        // 用我们的内存版而不是 mock
        InMemoryDatabase db;
        Lock lock;
        UpdateBug update_bug{db, lock};

        update_bug.execute(bug);
        // 对内存 db 用状态测试
        assert(db.getLastSaved() == bug);
    }
};

// 第二种方案：基于行为的测试（用 mock）。
class UpdateBugMockistTest {
public:
    void testBehaviorBasedTestingMockStyle() {
        Bug bug{"description"};
        auto db = std::make_unique<FakeDatabase>();
        auto lock = std::make_unique<FakeLock>();
        EXPECT_CALL(*lock, acquire());
        EXPECT_CALL(*db, save(bug));
        EXPECT_CALL(*lock, release());
        // 设 mock 少了两行。

        UpdateBug update_bug{*db, *lock};
        update_bug.execute(bug);
        // 验证它如预期发生
    }
};
```

我们需要对象各司一职，而这一职不是充当其他对象的服务定位器。使用 Guice 时，你能移除任何既有的服务定位器。当一个方法在它的主职责之外还充当定位器时，迪米特法则违反就发生了。

- 缺陷：`db.getLock()` 超出了 Database 的单一职责。它还违反迪米特法则，因为要用这个锁我们得调 `db.getLock().acquire()` 和 `db.getLock().release()`。
- 缺陷：测试 `UpdateBug` 类时，你得 mock 掉 `Database` 的 `getLock` 方法。
- 缺陷：`Database` 既当数据库，又当服务定位器（帮别人找锁）。它有身份危机。把迪米特法则违反和服务定位器行为合在一起，比任一问题单独出现都更糟。Database 的本意不是给别人分发其他服务的引用，而是把实体存进持久化存储。

Database 的 `getLock()` 方法应当被消除。即便 Database 需要持有一个锁的引用，更好的做法是 Database 不把它分享给别人。

*你永远不该去 mock 一个 setter 或 getter。*

这里给出两种方案：一种用基于状态的测试，另一种用基于行为的测试。第一种风格在对象被施以操作后针对其状态做断言。它不耦合到实现，只关心结果处于预期的状态。第二种风格用 mock 对象针对被测系统（SUT）的内部行为做断言。两种风格都成立，尽管不同的人对其中之一有强烈偏好。

### 问题：名叫"Context"的对象是去寻找违反的一大提示

```cpp
// 改造前：context 对象可以是个 std::map 或某种
// 自定义的大杂烩。

class MembershipPlan {
public:
    void processOrder(const UserContext& user_context) {
        const User& user = user_context.getUser();
        const PlanLevel& level = user_context.getLevel();
        const Order& order = user_context.getOrder();

        // ... 处理
    }
};
```

```cpp
// 改造后：用方法内真正需要的具体参数替换 context。

class MembershipPlan {
public:
    void processOrder(const User& user,
                      const PlanLevel& level,
                      const Order& order) {
        // ... 处理
    }
};
```

```cpp
// 改造前：一个对着糟糕 context 对象跑的测试示例。

void testWithContextMakesMeVomit() {
    MembershipPlan plan;
    UserContext user_context;
    user_context.setUser(User{"Kim"});
    PlanLevel level{143, "yearly"};
    user_context.setLevel(level);
    Order order{"SuperDeluxe", 100, true};
    user_context.setOrder(order);

    plan.processOrder(user_context);

    // 然后针对 user 等做断言……
}
```

```cpp
// 改造后：新设计更简单，也容易演进。

void testWithHonestApiDeclaringWhatItNeeds() {
    MembershipPlan plan;
    User user{"Kim"};
    PlanLevel level{143, "yearly"};
    Order order{"SuperDeluxe", 100, true};

    plan.processOrder(user, level, order);

    // 然后针对 user 等做断言……
}
```

context 对象理论上听着不错（改依赖不必改签名），但它们非常难测。

- 缺陷：你的 API 说测这个方法所需的一切就是一个 `userContext` map。可作为测试的编写者，你压根不知道它到底是什么！通常这意味着你写个测试传个 null 或空 map 进去，看它失败，再往 map 里逐步塞东西，直到它能过。
- 缺陷：有人可能声称这个 API"灵活"（因为加任何参数都不必改签名），但*它其实很脆弱*，因为你用不了重构工具；使用者不知道真正需要哪些参数。仅凭查看 API，无法确定它的合作者是谁。这让项目里的新人难以理解这个类的行为与目的。我们说这个 API 在它的依赖上撒了谎。

修复使用 context 对象的代码的方法，是用所需的具体对象替换它们。这会暴露真正的依赖，也可能帮你发现如何进一步分解对象，得到更好的设计。

## 这在何时不是缺陷：

### 注意（不成问题）：领域特定语言可以为了配置的便利违反迪米特法则

在某些领域特定语言（DSL）里工作，违反迪米特法则或许可以接受。它们为了配置的便利而违反迪米特法则。这不成问题，因为它是在以流式、易懂的方式构建一个值对象。一个例子是 Guice 模块里构建绑定时。

```cpp
// 改写自原文 Java 示例
// 一个 DSL 可能是可接受的违反。
// 即在 GUICE Module 的 configure 方法里
bind(Some.class)
    .annotatedWith(Annotation.class)
    .to(SomeImplementation.class)
    .in(SomeScope.class);
```
