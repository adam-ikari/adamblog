# 【文档翻译】编写可测试代码 · 缺陷四 · 类做的太多

> 译自 misko hevery 的 [Guide: Writing Testable Code](https://github.com/mhevery/guide-to-testable-code) 中的 *Flaw #4: Class Does Too Much*。为贴合 C++ 语境，代码示例由原文的 Java 改写而来，论述忠于原文。

一个类承担了太多职责。职责之间的交互被埋在类内部，你会把自己逼进死角——很难单独改动某一个职责。测试也找不到这些交互之间清晰的"接缝"。*依赖组件的构造（比如构建对象图）本身就是一项职责，应当从类真正的职责中剥离出去。用依赖注入传入预先配置好的对象。把单一职责提取成独立的类。*

> #### 危险信号
>
> - 概括这个类做什么时，用到了"和"这个字
> - 团队新成员读这个类、迅速"搞懂它"会是个挑战
> - 类有某些字段只在部分方法里用到
> - 类有只对参数进行操作的静态方法

## 为什么这是个缺陷

当类的职责和活动范围过大，你最终会得到这样的代码：

- 难以调试
- 难以测试
- 系统不可扩展
- 新人难以接手、难以交接
- 无法通过标准机制改变行为：装饰器、策略、子类化——你最后只能再加一个条件判断
- 难以给这个类起个名字来描述它做什么。每当在起名上犯难，代码就在告诉你：职责是混乱的。当对象有个清晰的名字，就容易让它保持聚焦、剥掉多余的包袱。

我们不希望代码库里有这样的代码。作为代码审查者，你有责任要求修改那些职责过多的类。

*这个缺陷也常被这样描述*

下面这些词没一个好听，却常常准确描述了承担过多职责的代码。即便一时不准，日久这些类也会长成这样：

- 厨房水槽（Kitchen Sink，什么都往里扔）
- 垃圾场（Dumping Ground）
- 行为里有太多"和"的类
- "首先要杀光所有 Manager"（*见莎士比亚*）
- 上帝类（God Class）
- "你可以看任何东西，除了这一个类"

## 识别这个缺陷

*试着用一句话概括一个类做什么。如果这句话里包含"和"，它大概就做得太多了。*

类的名字应当表达它做什么，应当清晰地传达它的角色。如果类需要 Manager、Utility、Context 这种大而化之的名字，它大概做得太多了。当你能准确、完整地命名某物，你就更能隔离职责、创建聚焦的类。当你命名不了，它大概该是多个对象。

- 令人担忧的描述示例："KillerAppServer 包含 main()，负责解析命令行参数**和**初始化过滤器链与 servlet**和**为 Google Servlet Engine 映射 servlet**和**控制服务器主循环……"
- 令人担忧的描述示例："SyndicationManager 缓存 syndication**和**实现复杂的过期逻辑**和**执行 RPC 重新填充缺失或过期条目**和**维护每个用户的 syndication 统计。"（实际上，初始化合作者可能是一项独立职责，与把它们装配好之后真正要做的事是分开的。）

*你的类做得太多的危险信号：* 构造函数做实事（见缺陷一）

- 过度滚动
- 许多字段
- 许多方法
- 无关/相关方法扎堆
- 许多合作者（传入的、在类内构造的、通过静态访问的、或从别的合作者里拽出来的）
- 这个类就是"感觉太大"。很难一次性装进脑子里、当作单一行为块来理解。让一个新人读这个类然后"秒懂"，会很难。
- 类与"哑合作者"打交道，这些合作者自己没有任何行为
- 公开方法背后隐藏着交互

*这个类在和"哑合作者"打交道，这些合作者自己没有行为。*

> 如果你正在合作的这个类替与它交互的合作者干了所有的活，那这份职责本该属于那些合作者。人们可能管这叫"上帝类"——它把本该在与之交互的其他对象里的行为全揽了过来。这常常由"基本类型偏执"（Primitive Obsession）引起。
>
> - [*Refactoring*, Fowler]

*隐藏的交互*藏在某个公开 API 背后，本可以通过组合更好地处理。

- 从设计角度看，这隐藏了一个组合本可提供的灵活性点。
- 为单一职责封装行为，通常是好的信息隐藏。
- 为职责之间的交互做封装（让一个类承担许多职责），通常是不好的。这会让协作变得脆弱。
- 从测试角度看，你得不到想要的接缝，无法孤立地测试各个部分。

## 修复这个缺陷

*如果这次变更引入了一个做得太多的类，要求作者把它拆开。*

1. 识别出各个独立的职责。
2. 给每一个起个干脆的名字。
3. 把功能提取到独立类中，每个职责一个类。
4. 可能有一个类承担隐藏的职责：在其余类之间居中调停。
5. 庆祝一下：现在你能比以前容易得多地孤立测试每个类了。

*如果处理的是一个此前就做得太多的遗留类，而你今天没法把整个遗留问题都修掉，你至少可以：*

1. 萌芽一个新类，唯一职责就是新功能。
2. 在你改动现有行为的地方提取一个类。当你着手现有功能时（比如再加一个条件判断），提取一个类把那份职责带走。这会从遗留类身上一块块地剥下来，你就能孤立测试每一块（用依赖注入）。

> 引入这些其他合作者时，你可能会发现组合显得别扭或不自然。尽管别扭，你今天也得迈出一步，防止这个大类继续膨胀。不这么做，它只会越长越大，揽进越来越多无关职责，越来越糟。

*如果这次变更里的类有某些字段只在少数方法里用到：*

- 如果你有几个方法是某个字段的唯一客户——那就是你的新类。把这些方法做的工作封装进一个新类。

*如果这次变更里有对参数进行操作的静态方法：*

- 静态方法常常是无家可归方法的信号。看看传进静态方法的参数。你大概有一个方法本该属于某个参数，或属于某个参数的包装器。把这个方法挪到它该属于的参数上。

一个 C++ 的例子，把无家可归的静态方法挪到它该在的对象上：

```cpp
// 改造前：静态方法对参数操作，像个孤儿
// 调用处：OrderUtils::calculateTotal(order) —— order 只是被动参数
class OrderUtils {
public:
    [[nodiscard]] static double calculateTotal(const Order& order) {
        double total = 0;
        for (const auto& item : order.items()) {
            total += item.price() * item.quantity();
        }
        return total;
    }
};
```

```cpp
// 改造后：方法归位到 Order 上，Order 有了自己的行为
// 调用处：order.total() —— 行为与数据在一起，也更好测试
class Order {
public:
    [[nodiscard]] double total() const {
        double sum = 0;
        for (const auto& item : items_) {
            sum += item.price() * item.quantity();
        }
        return sum;
    }

    [[nodiscard]] const std::vector<Item>& items() const { return items_; }

private:
    std::vector<Item> items_;
};
```

## 遗留：与缺陷共处

有些遗留类"超出了这一次变更的范围"。要求作者为了一个小改动去修一个巨大的既有问题，可能不合理。但划一条线、要求作者朝正确方向迈步而非让状况更糟，是合理的。比如，萌芽一个新类，而不是往一个既有的、难测的对象上再加一个职责。
