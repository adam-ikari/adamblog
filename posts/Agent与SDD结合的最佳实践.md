# 用 Agent 跑 SDD：七阶段瀑布流程最佳实践

## 前言

先纠正一个普遍误解：Agent 不是"会写代码的聊天机器人"，把它当聊天机器人用，是今天 AI 编程翻车的头号原因。你跟它说一句"帮我做个用户管理模块"，它写得挺欢，但你拿到的多半是没接需求、没过设计、测不了、接不上的半成品。

真正能让 Agent 稳定产出可用代码的，是把它摁进一条**有阶段、有门槛、有产出物**的流水线里。这条流水线我们其实早就有，就是瀑布模型的七阶段：**RA → SD → PD → CD → UT → IT → QA**。而 SDD（Spec-Driven Development，规格驱动开发）做的事，是让每个阶段的产出物——不是脑子里的想法，是落了地的规格文档——成为下一阶段的输入和约束。

这篇就按这七个阶段逐个讲：每阶段 Agent 该干什么、不该干什么，准入门槛和终点资产怎么用规格卡住它，以及踩过的坑。整个机制的核心就一句话：**Agent 擅长执行明确的规格，瀑布的纪律负责把规格明确出来**。两者咬合，AI 编程才能从"碰运气"变成"可工程化"。

## 先把三者的关系理清

很多人把 SDD、瀑布、Agent 当三件孤立的事，其实它们天然咬合：

- **瀑布**给的是**阶段纪律**——前后顺序、准入门槛、评审、受控基线。它解决"别乱跑"。
- **SDD**给的是**驱动方式**——每个阶段的终点是一份规格文档，这份文档不是写完就扔的备忘，而是驱动下一阶段机器可读的输入。它解决"别凭脑子传话"。
- **Agent**给的是**执行能力**——它擅长照着规格把活儿干了，但不擅长自己定义规格。它解决"别让人手动干重复活"。

咬合点就在每阶段的产出物上。没有瀑布的门槛，Agent 会跳过设计直接写代码；没有 SDD 的文档化，Agent 拿不到可执行的输入；没有 Agent 的执行，SDD 写一堆规格最后还是人手敲代码，瀑布又退化成又慢又重的人海战术。三者缺一，这套就跑不起来。

下面这套对照表是全文的骨架，后面七节就是逐行展开它：

| 阶段 | 关键动作 | 准入门槛 | 终点资产（驱动下一阶段） |
| --- | --- | --- | --- |
| RA 需求分析 | 定义功能、性能、接口需求 | 项目立项 | 需求规格说明书（SRS） |
| SD 系统设计 | 架构、模块划分、DB 设计 | SRS 评审通过 | 概要设计文档（HLD） |
| PD 程序设计 | 伪代码、类图、逻辑流程 | HLD 评审通过 | 详细设计文档（LLD） |
| CD 编码 | 代码实现、静态扫描 | LLD 评审通过 | 源代码 |
| UT 单元测试 | 函数/类逻辑验证、路径覆盖 | 代码编译通过 | 单元测试报告 |
| IT 集成测试 | 模块接口联调、数据流转验证 | UT 通过 | 集成测试报告 |
| QA 质量保障 | 合规审计、质量度量、准出评审 | IT 通过 | QA 准出报告 + 受控基线包 |

> **为什么用瀑布而不是敏捷**：不是瀑布更先进，是 Agent 更吃确定性。敏捷靠人在迭代里边走边对齐，Agent 没有"边走边对齐"的能力——给它模糊输入，它输出也模糊，且不会自己回头问。瀑布把"想清楚"前置到每个阶段门槛上，恰恰补了 Agent 的这个短板。所以这里用瀑布，是工程选择，不是怀旧。

## RA：需求分析 —— 让 Agent 帮你把需求问清楚，但别让它定义需求

RA 的产出是《需求规格说明书》（SRS），它要回答"做什么、做到什么程度、不做什么"。这一步 Agent 的角色是**提问者和记录者**，不是决策者。

怎么用：

- 把业务方的原始诉求（往往是一段口语化的描述）丢给 Agent，让它**反向产出一份 SRS 草稿**：功能清单、性能指标、接口要求、边界外的内容。
- 关键动作是让 Agent **列缺口**——它读完后，主动指出"这段需求里没说清响应时间要求""没定义异常时怎么处理""权限模型是 RBAC 还是 ABAC 没讲"。这些缺口就是你要回去跟业务方确认的问题。

这里最容易犯的错，是让 Agent 自己"补全"缺口。它补得出来，但补的是它的猜测，不是业务真实意图。**Agent 发现缺口是本事，替你做决策是越界。** 缺口必须由人确认后，再让 Agent 写进 SRS。

::: warning 准入与产出
准入门槛：项目立项。终点资产：评审通过的 SRS。SRS 没过评审，绝不让 Agent 进下一阶段——后面所有阶段都建立在它之上，地基歪了全歪。
:::

> **SDD 在这一步**：SRS 不是 Word 里一段散文，而是结构化的、可被后续阶段引用的规格。功能项要编号、性能要给可测的数值、接口要给契约。Agent 在后面阶段读它时，是按结构读的，不是按"大概意思"读的。

## SD：系统设计 —— 让 Agent 当你的架构师助理，拍板的还是你

SD 产出《概要设计文档》（HLD），定架构、模块划分、数据库设计、关键技术选型。准入是 SRS 评审通过。

这一步 Agent 能发挥的真实价值：

- **出方案备选**：给它 SRS，让它产出 2-3 套架构方案（如单体 vs 微服务、同步 vs 消息队列），各列利弊。注意，是让它**给选项和取舍**，不是让它**直接定一个**。
- **画图核验一致性**：让它把 HLD 里的模块依赖、数据流画成图，人眼看图比看文字更容易发现"这个模块谁都没调用""这条数据流断了"。
- **反向检查 SRS 覆盖**：让 Agent 拿 HLD 对照 SRS，逐条核对每个需求是否有设计承接，标出漏掉的需求。

陷阱在于技术选型。Agent 有偏好——它倾向推荐自己训练数据里更流行的方案，未必适合你的团队栈和约束。**选型必须人定**，Agent 只负责把选型理由写进 HLD，并标注替代方案。

::: warning 准入与产出
准入门槛：SRS 评审通过。终点资产：评审通过的 HLD。HLD 里定的模块边界和数据模型，是 PD 阶段细化设计的硬约束——PD 不能改模块边界，只能往里填细节。
:::

## PD：程序设计 —— Agent 真正发力的起点

PD 产出《详细设计文档》（LLD），定伪代码、类图、函数签名、逻辑流程。准入是 HLD 评审通过。

**这是 Agent 在整条链路里最该发力的阶段，却最常被跳过。** 很多人从 SD 直接跳到 CD 让 Agent 写代码，跳过 PD，结果 Agent 写出来的代码跟 HLD 对不上、互相接口打架、返工一轮接一轮。PD 就是来堵这个漏的。

怎么用：

- 给 Agent HLD，让它为每个模块产出 LLD：类结构、公开接口签名、核心算法的伪代码、异常处理路径。
- 让 Agent **做接口对齐**：A 模块调 B 模块，把两边的 LLD 拼一起，让 Agent 检查"参数类型对不对、返回值接不接得住、异常传不传"。这步人工做极枯燥，Agent 做又快又准。
- LLD 写完后，让 Agent **反向生成测试用例骨架**——还没写代码，但根据 LLD 就能推出该测哪些路径。这些用例骨架进 UT 阶段直接用。

PD 阶段的纪律是：**LLD 评审通过前，不许碰代码**。这条卡不住，SDD 就名存实亡——后面 Agent 写代码没有依据，又退化成聊天机器人。

::: tip 为什么 PD 最适合 Agent
PD 的活儿——把架构细化成可实现的接口和逻辑——高度结构化、规则明确、可机器核验，正是 Agent 的强项。而它又不像 RA 那样需要业务判断、不像 SD 那样需要架构权衡。把 Agent 的力气花在 PD，性价比最高。
:::

## CD：编码 —— Agent 照着 LLD 把代码敲出来

CD 产出源代码。准入是 LLD 评审通过。到这一步，前面三阶段的规格已经把"写什么"定死了，Agent 的工作终于变成纯执行。

怎么用：

- 把 LLD 喂给 Agent，让它**逐模块实现**，每完成一个模块立即过静态扫描。
- **不让 Agent 自由发挥**：LLD 没定义的工具函数、没列出的依赖、没要求的抽象层，一律不加。Agent 爱过度设计，给它 LLD 当紧箍咒。
- 代码写完，让 Agent **回贴 LLD**——把每个函数的实现对照 LLD 里的伪代码，标注偏离。有偏离要么是 LLD 没考虑全（回去补 LLD 再重写），要么是 Agent 自作主张（打回去）。

CD 阶段最大的坑是**规格漂移**：Agent 写着写着偏离了 LLD，人没盯住，等 UT 阶段才发现代码和设计对不上，这时返工成本已经很高。解法是把 CD 切成小颗粒——一个模块一交付、一核对，别让 Agent 一次性写一大片。

::: warning 准入与产出
准入门槛：LLD 评审通过。终点资产：通过静态扫描的源代码。CD 的产出是 UT 的输入，代码编译不过，UT 无从谈起。
:::

## UT：单元测试 —— 用 PD 阶段攒的用例骨架兜底

UT 产出《单元测试报告》。准入是代码编译通过。这一步的用例，PD 阶段就该攒好骨架了，UT 阶段是填充和执行。

怎么用：

- 把 PD 阶段 Agent 生成的测试用例骨架拿来，让它**补全成可跑的测试代码**，覆盖正常路径、边界值、异常路径。
- 跑测试，让 Agent **分析失败用例**：是代码 bug（回去 CD 修代码），还是 LLD 漏了路径（回去 PD 补 LLD），还是用例本身写错（修用例）。这三种原因要分清，别一律改代码糊弄过去。
- 出《单元测试报告》：覆盖率、通过率、未覆盖路径及原因。

UT 阶段的纪律是**覆盖率门槛**：没达到约定的覆盖率（比如核心逻辑 80%），不许进 IT。这条门槛是瀑布给的护栏，防止"差不多就行"的心态往后传。

> **SDD 在这一步**：UT 的判断标准来自 LLD——一个函数该有什么行为，LLD 里写了，测试就照着写。没有 LLD，UT 就变成"测 Agent 写出来的东西"，那是验算不是验证，测不出设计缺陷。

## IT：集成测试 —— 验的是模块拼起来还对不对

IT 产出《集成测试报告》。准入是 UT 通过。UT 验的是单个模块，IT 验的是模块拼起来——接口联调、数据流转、跨模块的端到端路径。

怎么用：

- 让 Agent 根据 SRS 和 HLD **生成集成测试场景**：哪些业务流程要跨模块走通、哪些接口要联调、哪些数据状态要流转。
- Agent 执行联调，重点抓**接口契约不一致**——A 模块传给 B 的字段，B 接不住。这类问题 UT 阶段发现不了（UT 不跨模块），到 IT 才暴露，是七阶段里返工最痛的环节。
- 失败时让 Agent 定位是哪段契约坏了，回 PD 修 LLD 还是回 CD 修代码，**别让 Agent 自己在代码层胡乱打补丁**——接口不一致往往是设计问题，不是代码 bug。

IT 通过，意味着系统作为一个整体能跑通业务流程了。这一关不过，别进 QA。

## QA：质量保障 —— 最后的准出闸门，别让 Agent 自己盖章

QA 产出《QA 准出报告》和受控基线包。准入是 IT 通过。这一步是合规审计、质量度量、准出评审，是整条链路的终点闸门。

怎么用：

- Agent 负责**出度量数据**：代码覆盖率、缺陷密度、需求追溯矩阵（每个 SRS 需求项有没有对应的设计、代码、测试）。
- Agent 负责**生成基线包**：源码、文档、测试报告、配置项打包，受控归档。
- **准出评审必须人做**。这是底线。Agent 可以把所有证据摆出来，但"能不能上"这个判断不能交给它——它有"乐观"倾向，倾向于报喜不报忧。准出签字是人负的责任，不是 Agent 能替的。

::: warning 最关键的纪律
整条链路里，能交给 Agent 的是执行，不能交的是**评审和准出**。RA 的 SRS 评审、SD 的 HLD 评审、PD 的 LLD 评审、QA 的准出，这四个门槛必须人过。Agent 可以辅助准备评审材料，但不能替你点头。一旦让 Agent 自己给自己评审，规格就形同虚设，瀑布退化为"Agent 写完就算"。
:::

## 贯穿全程的三条铁律

七阶段展开完，有三件事不是某个阶段的，是贯穿全程的，单独拎出来说：

**一是规格文档必须是结构化的，不是散文。** SRS、HLD、LLD 都要可被 Agent 机器读取——功能编号、接口契约、可测的性能数值。写成 Word 里一段段"系统应该能够…"的散文，Agent 读不进去，SDD 就落不了地。结构化是 SDD 区别于"写文档走形式"的根本。

**二是每个阶段都要留可追溯的证据链。** SRS 的需求项 → HLD 的模块 → LLD 的函数 → CD 的代码 → UT 的用例 → IT 的场景，这条链要能正反向查。Agent 最适合干这种"对账"的活——让它定期跑追溯检查，发现"某个需求没人实现""某段代码没有对应设计"的断裂点。需求追溯矩阵不是 QA 阶段才补的作业，是全程实时维护的状态。

**三是 Agent 的上下文要按阶段隔离。** 别让一个 Agent 会话从 RA 一路干到 QA——上下文越长越糊，它会忘了前面阶段的约束。每个阶段开新的会话，把该阶段的准入文档（上一阶段产出）作为输入喂进去，做完产出本阶段文档，会话结束。这恰恰对应瀑布的阶段隔离思想：每个阶段有明确的输入、输出、边界，互不串味。

## 总结

用 Agent 跑 SDD 瀑布，本质是把传统软件工程里"人传话、人评审、人执行"的链路，换成"规格传话、人评审、Agent 执行"。人从执行者变成规格制定者和门槛把关者，Agent 干它最擅长的——照规格执行。

这套真正管用的地方，不在 Agent 有多强，在**门槛有多硬**。SRS 没评审不许进 SD、LLD 没评审不许进 CD、准出必须人签——这些门槛一旦松了，Agent 的不确定性就会一层层往后漏，到 QA 才爆出来，返工成本是 RA 阶段的几十倍。

所以落实这套，别先盯着 Agent 怎么调，先盯门槛怎么卡。把七张准入门票捏在手里，Agent 就是你最听话的执行机器；放任它跨阶段跑，它就是最贵的返工源头。
