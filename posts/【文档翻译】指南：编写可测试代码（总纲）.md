# 【文档翻译】指南：编写可测试代码（总纲）

_为代码编写测试代码是每个软件工程师的美德。_

> 译自 misko hevery 的 [Guide: Writing Testable Code](https://github.com/mhevery/guide-to-testable-code)。本篇是总纲，列出可测试代码的四大缺陷及危险信号；每条缺陷的深入论述见系列后续各篇。代码示例由原文的 Java 改写为 C++。

## 前言

很多程序员都有写的代码难以测试的痛苦，这其实说明了代码某些设计的不合理。谷歌的实践中很早就发现了某些代码中不好的设计，正是这些设计阻碍了测试代码的编写。

下面是阻碍可测试性的四大缺陷。每一条都先列出危险信号，再附一句简要说明；要真正讲透识别与修复，见各缺陷的专篇。

### 缺陷 1：构造函数做了实际工作

详见 [缺陷一：构造函数做实事](/posts/【文档翻译】编写可测试代码-缺陷一-构造函数做实事)。

**危险信号**：

- 构造函数中出现 `new` 关键字
- 构造函数调用静态方法
- 构造函数中出现除了字段初始化之外的任何操作
- 构造函数未能将对象完全初始化（请注意 `initialize` 方法）
- 构造函数中出现控制流程（例如条件或循环逻辑）
- 构造函数中构建复杂对象，而不是使用工厂或构建器
- 添加或使用初始化块

**要害**：构造函数本该只做字段赋值，把对象装配好。一旦它干了"实事"——创建合作者、调静态方法、跑控制流——测试时就没法替换这些合作者，对象也无法在未就绪的状态下被构造出来供测试。

### 缺陷 2：挖掘合作者类

详见 [缺陷二：挖掘合作者](/posts/【文档翻译】编写可测试代码-缺陷二-挖掘合作者)。

**危险信号**：

- 传递了对象，但从未直接使用（仅用于访问其他对象）
- 违反了德米特定律：方法调用链在一个带有多个点（`.`）的对象图中移动
- 可疑的名称：`context`、`environment`、`principal`、`container` 或 `manager`

**要害**：一个对象应该只跟它直接需要的合作者打交道。如果它拿了一个对象只为去够另一个对象（`a.getB().getC().doSomething()`），说明它真正依赖的是链条尽头的那个，中间的都是负担——测试时要装配整条链，脆弱又麻烦。

### 缺陷 3：脆弱的全局状态和单例

详见 [缺陷三：全局状态与单例](/posts/【文档翻译】编写可测试代码-缺陷三-全局状态与单例)。

**危险信号**：

- 添加或使用单例模式
- 添加或使用静态字段或静态方法
- 添加或使用静态初始化块
- 添加或使用注册表
- 添加或使用服务定位器

**要害**：全局状态让对象的依赖隐而不显——测试无法隔离，因为谁都能改那个全局变量，测试之间互相污染。单例看似优雅，实则是披着面向对象外衣的全局状态，同样难以测试。

### 缺陷 4：类做的太多

详见 [缺陷四：类做的太多](/posts/【文档翻译】编写可测试代码-缺陷四-类做的太多)。

**危险信号**：

- 总结类的职责中包含"和"这个关键字
- 对于团队的新成员来说，阅读和掌握一个类成为挑战
- 类具有仅在某些方法中使用的字段
- 类具有仅对参数进行操作的静态方法

**要害**：一个类承担过多职责，职责间的交互埋在类内部，测试找不到清晰的接缝。能用一句话（不含"和"）说清一个类做什么，它才是聚焦的；说不清，就该拆。
