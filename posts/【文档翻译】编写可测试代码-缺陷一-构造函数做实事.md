# 【文档翻译】编写可测试代码 · 缺陷一 · 构造函数做实事

> 译自 misko hevery 的 [Guide: Writing Testable Code](https://github.com/mhevery/guide-to-testable-code) 中的 *Flaw #1: Constructor does Real Work*。为贴合 C++ 语境，代码示例由原文的 Java 改写而来，论述忠于原文。

构造函数里做实事——比如创建/初始化合作者、与其他服务通信、设置自身状态的逻辑——*会抹掉测试所需的接缝*，迫使子类/mock 继承不想要的行为。构造函数里干太多活，会妨碍你在测试中实例化对象或替换合作者。

> #### 危险信号
>
> - 构造函数里或字段声明处出现 `new` 关键字
> - 构造函数里或字段声明处调用静态方法
> - 构造函数里做了除字段赋值以外的任何事
> - 构造函数结束后对象没有完全初始化（当心 `initialize` 方法）
> - 构造函数里有控制流（条件判断或循环逻辑）
> - 在构造函数里构造复杂的对象图，而不是用工厂或 builder
> - 添加或使用初始化块

## 为什么这是个缺陷

当你的构造函数必须实例化并初始化它的合作者，结果往往是一个僵硬、过早耦合的设计。这样的构造函数在测试时关掉了注入测试合作者的能力。

*它违反了单一职责原则*

当合作者的构造与初始化混在一起，意味着这个类只有一种配置方式，从而关掉了原本可用的复用机会。对象图的创建是一项完整的职责——和这个类存在的本来目的不是一回事。在构造函数里干这种活，违反了单一职责原则。

*直接测试很困难*

测试这样的构造函数很困难。要实例化一个对象，构造函数必须执行。而如果构造函数干很多活，你在测试里创建对象时就被迫要做这些活。如果合作者访问外部资源（比如文件、网络服务或数据库），合作者里的细微变化可能需要反映到构造函数里，却可能因为缺少测试覆盖而被漏掉——而测试覆盖之所以缺失，又是因为构造函数太难测、测试根本没写。我们陷入了恶性循环。

*靠子类化并重写来测试仍然是个缺陷*

另一种情况：构造函数自己没干多少活，但委托给一个预期在测试子类里被重写的方法。这或许能绕开构造困难的问题，但"子类化来测试"这种把戏只能作为最后手段。另外，靠子类化，你就测不到你重写的那个方法。而那个方法干很多活（记住——这正是它当初被创建出来的原因），所以它大概应该被测试。

*它把合作者强加给你*

有时候你测试一个对象时，并不想真的创建它所有的合作者。比如你不会想要一个真的会跟 MySQL 服务通信的 `MySqlRepository` 对象。然而，如果在被测系统（SUT）内部用 `new MySqlRepositoryServiceThatTalksToOtherServers()` 直接创建它们，那你就被迫要用这个重量级对象。

*它抹掉了一个"接缝"*

接缝是你可以切入代码库、移除依赖、实例化小而聚焦的对象的地方。当你在构造函数里 `new XYZ()`，你永远没法得到一个不同的（子类）对象被创建。（更多关于接缝的内容，见 Michael Feathers 的 *Working Effectively with Legacy Code* 一书。）

*即便你有多个构造函数（有些是"仅用于测试"的），它仍然是个缺陷*

单独搞一个"仅用于测试"的构造函数并不解决问题。那些干活的构造函数还是会被其他类使用。即便你能孤立地测试这个对象（用测试专用的构造函数创建它），你迟早会撞上其他类用了那个难测的构造函数。而测试那些类时，你的手脚就被绑住了。

*底线*

一切归结为：孤立地构造这个类，或*用测试替身合作者*构造它，到底有多难或多容易。

- 如果难，就是你在构造函数里干了太多活！
- 如果容易，那给自己拍个背。

写代码时始终想一想：测试这个对象会有多难？用你正在写的这个构造函数实例化它会容易吗？（记住，你的测试类不会是实例化这个类的唯一代码。）

> 太多设计充斥着"*实例化其他对象、或从全局可访问位置获取对象的对象。这些编程实践若不加节制，会导致高度耦合、难以测试的设计。*"
>
> - [J.B. Rainsberger, *JUnit Recipes*, Recipe 2.11]

## 识别这个缺陷

检查这些症状：

- `new` 关键字构造了任何你在测试里想用测试替身替换的东西吗？（通常是任何比简单值对象更大的东西。）
- 有任何静态方法调用吗？（记住：静态调用不可 mock、不可注入，所以一旦看到 `Server.init()` 或类似的东西，你脑子里警报就该响起来！）
- 有任何条件或循环逻辑吗？（每次实例化对象你都得成功穿越这段逻辑。这不仅在你直接测试这个类时导致大量 setup 代码，而且在你测试任何相关类、恰好需要这个类时也一样。）

写代码或审查代码时，思考一个根本问题：

> 我要怎么测这个？
>
> *"如果答案不显然，或者看起来测试会很丑、很难写，那就把它当成一个警告信号。你的设计大概需要修改；改来改去直到代码容易测试，你的设计也会因这番努力变得好得多。"*
>
> - [Hunt, Thomas. *Pragmatic Unit Testing in Java with JUnit*, p 103（有点过时，但值得一读且很快能读完）]

> **注意**：
> 构造**值对象**在许多情况下是可以接受的（例子：LinkedList、HashMap、User、EmailAddress、CreditCard）。值对象的关键特征是：(1) 构造起来是平凡的 (2) 以状态为中心（大量 getter/setter，行为很少）(3) 不引用任何服务对象。

## 修复这个缺陷

*不要在构造函数里创建合作者，而是把它们传进来*

把对象图构造和初始化的职责移到另一个对象里。（比如提取一个 builder、工厂或 Provider，把这些合作者传给你的构造函数。）

例子：如果你依赖一个 `DatabaseService`（希望它是个接口），那就用依赖注入（DI）把你需要的那个确切的 `DatabaseService` 子类对象传进构造函数。

重复一遍：*不要在构造函数里创建合作者*，而是把它们传进来。（别去找东西！去要东西！）

如果传进来的对象还需要做些初始化，你有三个选择：

- *使用 Guice 的最佳实践：*用 `Provider<YourObject>` 来创建并初始化 `YourObject` 的构造参数。把对象初始化和对象图构造的职责留给 Guice。这会消除边走边初始化对象的需要。有时你会在 Provider 之外还用 Builder 或工厂，然后把这些 builder 和工厂传进构造函数。

- *使用手动依赖注入的最佳实践：*为 `YourObject` 的构造参数用一个 Builder 或工厂。通常一个工厂对应一整个对象图，见下方例子。（所以你不必担心每个类一个工厂导致类爆炸。）工厂的职责是创建对象图、不做任何活。（你在工厂里应该看到的全是一堆 `new` 关键字和传来传去的引用。）对象图的职责是干活、不做对象实例化（应用逻辑类里应该严重缺少 `new` 关键字）。

- *仅作最后手段：*在你的类里有一个 `init(...)` 方法，可在构造之后调用。能避免就避免，优先用另一个对象——它的唯一职责是为这个对象配置参数。（如果用 Guice，那通常是个 Provider。）

## 具体代码示例：改造前后

从根本上说，"构造函数做实事"指的是任何让*实例化你的对象变困难*、或*引入测试替身对象变困难*的事。

### 问题：构造函数里或字段声明处出现 "new" 关键字

```cpp
// 改造前：在类的构造函数里直接调用 new。
// （永远堵死了创建不同 kitchen、bedroom 合作者的接缝。）

class House {
public:
    House() {
        bedroom_ = std::make_unique<Bedroom>();
    }

    // ...

private:
    Kitchen kitchen_{std::make_unique<Kitchen>()};  // 字段声明处直接 new
    std::unique_ptr<Bedroom> bedroom_;
};
```

```cpp
// 改造后：让外部把对象创建好、传进来。

class House {
public:
    House(std::unique_ptr<Kitchen> k, std::unique_ptr<Bedroom> b)
        : kitchen_(std::move(k)), bedroom_(std::move(b)) {}

    // ...

private:
    std::unique_ptr<Kitchen> kitchen_;
    std::unique_ptr<Bedroom> bedroom_;
};
```

```cpp
// 改造前：一次变得相当难的测试

class HouseTest {
public:
    void testThisIsReallyHard() {
        auto house = std::make_unique<House>();
        // 糟了！我被构造函数里创建的
        // Kitchen 和 Bedroom 对象绑死了。

        // ...
    }
};
```

```cpp
// 改造后：新版极易测试，可用任意测试替身合作者。

class HouseTest {
public:
    void testThisIsEasyAndFlexible() {
        auto dummy_kitchen = std::make_unique<DummyKitchen>();
        auto dummy_bedroom = std::make_unique<DummyBedroom>();

        auto house = std::make_unique<House>(
            std::move(dummy_kitchen), std::move(dummy_bedroom));

        // 太棒了，我可以用更轻量的测试替身。

        // ...
    }
};
```

这个例子把对象图创建和逻辑混在了一起。测试中我们常想创建一个和生产线不同的对象图。通常是个更小的图，部分对象被测试替身替换。把 `new` 留在原地，我们永远没法为测试创建对象图。见："How to think about the new operator"。

- 缺陷：字段声明处直接实例化对象，和构造函数里干活有同样的问题。
- 缺陷：这个或许容易实例化，但如果 `Kitchen` 代表某种昂贵的东西（比如文件/数据库访问），它就不好测，因为我们永远没法把 `Kitchen` 或 `Bedroom` 换成测试替身。
- 缺陷：你的设计更脆，因为你永远没法在 `House` 里多态地替换 kitchen 或 bedroom 的行为。

如果 `Kitchen` 是个值对象，比如 LinkedList、Map、User、Email Address 等，那只要这些值对象不引用服务对象，我们就可以原地创建。服务对象是最可能需要用测试替身替换的那类，所以你永远不想用直接实例化或静态方法调用把它锁死。

### 问题：构造函数接收一个未完全初始化的对象、得自己去配置它

```cpp
// 改造前：SUT 初始化合作者。这妨碍了测试和
// Garden 的使用者去改变它们。

class Garden {
public:
    Garden(std::shared_ptr<Gardener> joe)
        : joe_(std::move(joe)) {
        joe_->setWorkday(std::make_shared<TwelveHourWorkday>());
        joe_->setBoots(std::make_shared<BootsWithMassiveStaticInitBlock>());
    }

    // ...

private:
    std::shared_ptr<Gardener> joe_;
};
```

```cpp
// 改造后：让外部创建 gardener，由一个
// 工厂/Provider 来配置它。

class Garden {
public:
    explicit Garden(std::shared_ptr<Gardener> joe)
        : joe_(std::move(joe)) {}

    // ...

private:
    std::shared_ptr<Gardener> joe_;
};

// 在装配对象图的地方（工厂 / Provider）

[[nodiscard]] std::shared_ptr<Gardener> makeGardenerJoe(
    std::shared_ptr<Workday> workday,
    std::shared_ptr<BootsWithMassiveStaticInitBlock> bad_boots) {
    auto joe = std::make_shared<Gardener>();
    joe->setWorkday(workday);

    // 理想情况下，你会重构掉那个静态初始化。
    joe->setBoots(bad_boots);
    return joe;
}
```

```cpp
// 改造前：一个很慢、且被迫多次运行
// 静态初始化块的测试。

class GardenTest {
public:
    void testMustUseFullFledgedGardener() {
        auto gardener = std::make_shared<Gardener>();
        auto garden = std::make_unique<Garden>(gardener);

        AphidPlague(garden).infect();
        garden->notifyGardenerSickShrubbery();

        assert(gardener->isWorking());
    }
};
```

```cpp
// 改造后：新测试跑得快，不依赖那个慢的
// BootsWithMassiveStaticInitBlock。

class GardenTest {
public:
    void testUsesGardenerWithDummies() {
        auto gardener = std::make_shared<Gardener>();
        gardener->setWorkday(std::make_shared<OneMinuteWorkday>());
        // 传 nullptr 也可以，因为这个测试里用不到。
        gardener->setBoots(nullptr);

        auto garden = std::make_unique<Garden>(gardener);

        AphidPlague(garden).infect();
        garden->notifyGardenerSickShrubbery();

        assert(gardener->isWorking());
    }
};
```

对象图创建（为 `Garden` 创建并配置 `Gardener` 合作者）是一项不同于 `Garden` 该干什么的职责。当配置和实例化在构造函数里混在一起，对象变得更脆、绑死在具体的对象图结构上。这让代码更难修改，也（或多或少）不可能测试。

- 缺陷：`Garden` 需要一个 `Gardener`，但配置 gardener 不该是 `Garden` 的职责。
- 缺陷：在 `Garden` 的单元测试里，工作时长被构造函数硬设成 12 小时，迫使我们让 Joe 干 12 小时。这种强加的依赖会让测试变慢。单元测试里你会想传入一个更短的工作时长。
- 缺陷：你没法换靴子。你大概想给靴子用个测试替身，以避开加载和使用 `BootsWithMassiveStaticInitBlock` 的麻烦。（静态初始化块常常危险且棘手，尤其当它们和全局状态交互时。）

当你需要初始化合作者时，用两个对象：先把它们初始化好，再把完全初始化好的对象传进目标类的构造函数。

### 问题：构造函数里违反迪米特法则

```cpp
// 改造前：违反迪米特法则
// 因为依赖过多而脆弱
// 把对象查找和赋值混在一起

class AccountView {
public:
    AccountView() {
        user_ = RPCClient::getInstance().getUser();
    }

private:
    User user_;
};
```

```cpp
// 改造后：只传入直接需要的东西

class AccountView {
public:
    explicit AccountView(User user)
        : user_(std::move(user)) {}

private:
    User user_;
};

// User 由工厂/Provider 提供
[[nodiscard]] User getUser(RPCClient& rpc_client) {
    return rpc_client.getUser();
}

// RPCClient 也由 Provider 提供，且不再是单例
[[nodiscard]] std::shared_ptr<RPCClient> getRPCClient() {
    // 我们移除了 JVM 单例，
    // 改由容器管理作用域。
    return std::make_shared<RPCClient>();
}
```

```cpp
// 改造前：难测，因为需要真的 RPCClient

class AccountViewTest {
public:
    void testUnfortunatelyWithRealRPC() {
        auto view = std::make_unique<AccountView>();
        // 唉！只能连一个真的 RPCClient。
        // 这个测试现在很慢。

        // ...
    }
};
```

```cpp
// 改造后：用依赖注入，很容易测

class AccountViewTest {
public:
    void testLightweightAndFlexible() {
        User user = DummyUser{};
        auto view = std::make_unique<AccountView>(std::move(user));
        // 用测试替身 user，好测又快。

        // ...
    }
};
```

这个例子里我们伸进了应用的全局状态，抓住了 `RPCClient` 单例。结果我们并不需要这个单例，我们只想要 `User`。第一：我们在干活（针对静态方法，而静态方法零接缝）。第二：这违反了"迪米特法则"。

- 缺陷：我们没法轻易拦截那个调用，好让它返回一个 mock `RPCClient` 来测试。（静态方法不可拦截、不可 mock。）
- 缺陷：既然被测类不需要 `RPCClient`（它没把 rpc 实例存进字段），为什么为了测试要 mock 掉 `RPCClient`？我们只需要持久化 `User`。
- 缺陷：每个需要构造 `AccountView` 类的测试都得面对上面这些问题。即便我们为一个测试解决了这些问题，也不想在其他测试里再解决一遍。比如另一个类可能需要 `AccountView`。于是在那个类的测试里我们也得成功穿越这个构造函数。

改进后的代码只传入了直接需要的东西：`User` 合作者。测试时你只需创建一个（真的或测试替身）`User` 对象。这让设计更灵活，*也*让可测试性更好。

我们用容器来提供一个来自 `RPCClient` 的 User。在单元测试里我们不用容器，而是直接创建 `User` 和 `AccountView`。

### 问题：构造函数里创建不需要的第三方对象

```cpp
// 改造前：创建不需要的第三方对象，
// 把对象构造和逻辑混在一起，
// "new" 关键字抹掉了用其他
// EngineFactory 来测试的接缝。
// 也把你绑在（慢的）文件系统上。

class Car {
public:
    explicit Car(const std::string& file_path) {
        std::string model = readEngineModel(file_path);
        engine_ = std::make_unique<EngineFactory>()->create(model);
    }

    // ...

private:
    std::unique_ptr<Engine> engine_;
};
```

```cpp
// 改造后：只请求它确切需要的东西

class Car {
public:
    explicit Car(std::unique_ptr<Engine> engine)
        : engine_(std::move(engine)) {}

    // ...

private:
    std::unique_ptr<Engine> engine_;
};

// 在工厂/Provider 里给出 Engine
[[nodiscard]] std::unique_ptr<Engine> getEngine(
    EngineFactory& engine_factory, const std::string& model) {
    return engine_factory.create(model);
}

// 在别处还有获取工厂和 model 的 Provider
```

```cpp
// 改造前：测试暴露了 Car 的脆弱

class CarTest {
public:
    void testNoSeamForFakeEngine() {
        // 啊！我讨厌在单元测试里用文件
        std::string file_path = "engine.config";
        auto car = std::make_unique<Car>(file_path);

        // 我想用个假 engine 测试，
        // 但做不到，因为 EngineFactory
        // 只会造真 engine。
    }
};
```

```cpp
// 改造后：现在能看到一个灵活、可注入的设计

class CarTest {
public:
    void testShowsWeHaveCleanDesign() {
        auto fake_engine = std::make_unique<FakeEngine>();
        auto car = std::make_unique<Car>(std::move(fake_engine));

        // 现在测试容易了，Car 只要了它确切需要的东西。
    }
};
```

从语言上说，要求一辆 `Car` 拿到一个 `EngineFactory` 去造自己的 engine 是说不通的。车该被给一个现成的 engine，而不是自己去想怎么造。你开去上班的那辆车不该有一个指回它工厂的引用。同理，有些构造函数会伸手去够并不直接需要的第三方对象——需要的只是那个第三方对象能创建出来的东西。

- 缺陷：传入一个文件，而最终需要的只是一个 Engine。
- 缺陷：创建一个第三方对象（`EngineFactory`）并承担这种不可注入、不可重写的创建所带来的任何附带成本。这让你的代码更脆，因为你没法换工厂、没法决定开始缓存它、也没法阻止它在创建新 `Car` 时运行。
- 缺陷：让 car 知道怎么造一个 `EngineFactory`、然后它又知道怎么造一个 engine，这很蠢。（不知怎的，当这些对象更抽象时，我们往往意识不到自己在犯这个错。）
- 缺陷：每个需要构造 `Car` 类的测试都得面对上面这些问题。即便我们为一个测试解决了，也不想在其他测试里再解决。比如 `Garage` 的另一个测试可能需要 `Car`。于是在 Garage 测试里我也得成功穿越 Car 构造函数。而且我还被迫创建一个新的 `EngineFactory`。
- 缺陷：每次调用 `Car` 构造函数，测试都得访问一个文件。这很慢，也妨碍测试成为真正的单元测试。

移除这些第三方对象，用简单的变量赋值替换构造函数里的活。在构造函数里把预先配置好的变量赋给字段。让另一个对象（工厂、builder 或容器 Provider）去做构造参数的实际构造。从你的主对象身上剥离对象图构造的职责，你会得到一个更灵活、更易维护的设计。

```cpp
        socket_ = std::make_unique<Socket>(FLAG_PORT::get());
```

```cpp
class PingServer {
```

```cpp
// Provider 的职责是给出一个完全配置好的 Socket，
// 这可能涉及比 "new" 更多的事。

[[nodiscard]] std::unique_ptr<Socket> getSocket(int port) {
```

```cpp
// 改造前：测试脆弱，直接绑在
// Flag 的静态方法（全局状态）上。

        auto server = std::make_unique<PingServer>();
        // 这看起来人畜无害，但实际
```

看起来像个简单的无参构造函数，其实有一大堆依赖。API 又在骗你，装作它容易创建，可实际上 `PingServer` 很脆、绑在全局状态上。

- 缺陷：你的测试得依赖全局变量 `FLAG_PORT` 才能实例化这个类。这会让测试不稳定，因为测试顺序会起作用。
- 缺陷：依赖一个静态访问的 flag 值，妨碍你并行跑测试。因为并行跑的测试可能同时改这个 flag 值，导致失败。
- 缺陷：如果 socket 需要额外配置（比如调某些方法），那做不到，因为对象构造发生在错误的地方。Socket 在 `PingServer` 内部创建，这是本末倒置。它该发生在外部，发生在一个唯一职责就是对象图构造的东西里——即一个容器 Provider。

`PingServer` 最终需要的是一个 socket，不是一个端口号。传入端口号，我们就得用真的 socket/线程测试。传入 socket，测试里就能造个 mock socket、不用任何真 socket/线程就能测这个类。显式传入端口号移除了对全局状态的依赖，大大简化测试。更好的是直接传入最终需要的那个 socket。

### 问题：构造函数里直接读 flag 并创建对象

```cpp
// 改造前：根据 flag 值分支来决定状态。

class CurlingTeamMember {
public:
    CurlingTeamMember() {
        if (FLAG_isSuedeJersey::get()) {
            jersey_ = std::make_unique<SuedeJersey>();
        } else {
            jersey_ = std::make_unique<NylonJersey>();
        }
    }

private:
    std::unique_ptr<Jersey> jersey_;
};
```

```cpp
// 改造后：我们把 Jersey 的选择职责
// 移到了一个 Provider 里。

class CurlingTeamMember {
public:
    // 推荐，因为构造/初始化的职责
    // 与这个对象在构造函数之外的职责
    // 已经分开。
    explicit CurlingTeamMember(std::unique_ptr<Jersey> jersey)
        : jersey_(std::move(jersey)) {}

private:
    std::unique_ptr<Jersey> jersey_;
};

// 用 FlagBinder（你写的一个类，知道怎么把
// 命令行 flag 绑到可注入参数）把 flag 类的
// 所有 flag 接进容器的可注入范围。

// 通过请求 Provider<SuedeJersey>
// 而不是调用 new SuedeJersey
// 你让 SuedeJersey 自由地去要它自己的依赖。
[[nodiscard]] std::unique_ptr<Jersey> getJersey(
    SuedeJerseyProvider& suede_provider,
    NylonJerseyProvider& nylon_provider,
    bool suede) {
    if (suede) {
        return suede_provider.get();
    }
    return nylon_provider.get();
}
```

```cpp
// 改造前：测试 CurlingTeamMember 很难。
// 实际上除了 SuedeJersey 或 NylonJersey
// 你用不了任何 Jersey。

class CurlingTeamMemberTest {
public:
    void testImpossibleToChangeJersey() {
        // 你被迫用全局状态。
        // ... 把 flag 设成你想要的样子
        auto russ = std::make_unique<CurlingTeamMember>();

        // 测试被锁死在用上面那两种 jersey 之一。
    }
};
```

```cpp
// 改造后：代码现在用了灵活的替代方案：
// 依赖注入。

class CurlingTeamMemberTest {
public:
    void testWithAnyJersey() {
        // 不用碰 flag
        auto jersey = std::make_unique<LightweightJersey>();
        auto russ = std::make_unique<CurlingTeamMember>(std::move(jersey));

        // 测试可以自由用任何 jersey。
    }
};
```

Guice 有个叫 `FlagBinder` 的东西，让你以很低的成本移除 flag 引用、换成注入的值。flag 被广泛用于改变运行时参数，但我们不必为了它们的全局状态去直接读 flag。

- 缺陷：直接读 flag 是伸手进全局状态取值。这不可取，因为全局状态不孤立：之前的测试可能把它设成了别的值，或别的线程可能意外改了它。
- 缺陷：根据 flag 值直接构造不同类型的 `Jersey`。你实例化这个类的测试没有接缝去注入一个不同的 Jersey 合作者。
- 缺陷：这个类的职责过宽：既有它本来的核心目的，现在又加上了 Jersey 配置。改成传入一个预先配置好的 `Jersey` 对象更好。另一个对象可以承担配置 Jersey 的职责。

用 FlagBinder（你写的一个类，知道怎么把命令行 flag 绑到可注入参数），把一个类里所有的 flag 都接到 Guice 的可注入范围里。

### 问题：把构造函数的"活"挪进一个 initialize 方法

```cpp
// 改造前：有静态、单例、还有一个棘手的
// initialize 方法，这个类很脆。

class VisualVoicemail {
public:
    // 用了容器注入
    explicit VisualVoicemail(User user)
        : user_(std::move(user)) {
        // 看我看我，骄傲不？我有个
        // 简单的构造函数，还用了 Guice
    }

    void initialize() {
        Server::readConfigFromFile();
        Server& server = Server::getSingleton();
        calls_ = server.getCallsFor(user_);
    }

    // 这招挺绕，但我觉得我搞明白怎么让它可测了！
    void setCalls(std::vector<Call> calls) {
        calls_ = std::move(calls);
    }

    // ...

private:
    User user_;
    std::vector<Call> calls_;
};
```

```cpp
// 改造后：用依赖注入和容器，
// 这是个更优的设计。

class VisualVoicemail {
public:
    explicit VisualVoicemail(std::vector<Call> calls)
        : calls_(std::move(calls)) {}
};

// 你需要一个 Provider 来获取 calls
[[nodiscard]] std::vector<Call> getCalls(Server& server, const User& user) {
    return server.getCallsFor(user);
}

// 以及一个 Server 的 Provider。容器也让你
// 能摆脱 JVM 单例。
[[nodiscard]] Server& getServer(ServerConfig& config) {
    static Server server{config};
    return server;
}

[[nodiscard]] ServerConfig getServerConfig(const std::string& path) {
    return ServerConfig{std::filesystem::path{path}};
}
```

```cpp
// 改造前：测试暴露了脆弱的设计

class VisualVoicemailTest {
public:
    void testExposesBrittleDesign() {
        User dummy_user = DummyUser{};
        auto voicemail = std::make_unique<VisualVoicemail>(std::move(dummy_user));
        voicemail->setCalls(buildListOfTestCalls());

        // 技术上这是能测的，前提是
        // 你不需要 Server 已经读过
        // 配置文件。但不测 initialize()
        // 就排除了重要行为。

        // 而且代码很脆，以后加新功能很难。
    }
};
```

```cpp
// 改造后：依赖注入暴露了你的依赖，
// 并提供了接缝去注入不同的合作者。

class VisualVoicemailTest {
public:
    void testFlexibleDesignWithDI() {
        auto voicemail = std::make_unique<VisualVoicemail>(
            buildListOfTestCalls());

        // ... 现在你想怎么测就怎么测。
    }

```

把"活"挪进 initialize 方法不是解决办法。你需要把对象解耦成单一职责。（其中一个单一职责就是提供一个完全配置好的对象图。）

- 缺陷：乍一看似乎 Guice 用得不错。测试 `VisualVoicemail` 对象构造起来很容易。然而代码仍然很脆，绑在好几个静态初始化调用上。
- 缺陷：initialize 方法是这个对象职责过多的显眼信号：`VisualVoicemail` 该干的事，加上初始化它的依赖。依赖初始化该发生在另一个类里，把*所有*准备就绪的对象传进构造函数。
- 缺陷：测试中若想调 initialize 方法，`Server::readConfigFromFile()` 是不可拦截的。
- 缺陷：测试中 Server 不可初始化。要用它，你就被迫从全局单例状态去拿。如果两个测试并行跑，或之前的测试用不同方式初始化了 Server，全局状态就会咬你一口。
- 缺陷：通常 `@VisibleForTesting` 注解是一种坏味道，说明这个类当初没被写成容易测试的样子。即便它能让你设置 calls 列表，那也只是绕过 `initialize()` 里根本问题的一个*补丁*。

解决这个缺陷，像许多其他缺陷一样，靠的是移除 JVM 强制的全局状态、使用依赖注入。

### 问题：有多个构造函数，其中一个只为测试而设

```cpp
// 改造前：一半容易构造，另一半昂贵。
// 而用那个昂贵构造函数的合作者——
// 它们自己也会变昂贵。

class VideoPlaylistIndex {
public:
    // 仅为测试：一个容易的构造函数！
    explicit VideoPlaylistIndex(std::shared_ptr<VideoRepository> repo)
        : repo_(std::move(repo)) {}

    VideoPlaylistIndex() {
        repo_ = std::make_shared<FullLibraryIndex>();
    }

    // ...

private:
    std::shared_ptr<VideoRepository> repo_;
};

// 一个合作者，因为硬编码的 index 构造
// 而变得昂贵。

class PlaylistGenerator {
public:
    Playlist buildPlaylist(const Query& q) {
        return index_.search(q);
    }

private:
    std::shared_ptr<VideoPlaylistIndex> index_ =
        std::make_shared<VideoPlaylistIndex>();
};
```

```cpp
// 改造后：容易构造，且没有其他对象
// 因使用昂贵构造函数而受害。

class VideoPlaylistIndex {
public:
    // 一个构造函数统治一切
    explicit VideoPlaylistIndex(std::shared_ptr<VideoRepository> repo)
        : repo_(std::move(repo)) {}

private:
    std::shared_ptr<VideoRepository> repo_;
};

// 一个合作者，现在也好构造了。

class PlaylistGenerator {
public:
    // 用手动 DI 传进来
    explicit PlaylistGenerator(std::shared_ptr<VideoPlaylistIndex> index)
        : index_(std::move(index)) {}

    Playlist buildPlaylist(const Query& q) {
        return index_->search(q);
    }

private:
    std::shared_ptr<VideoPlaylistIndex> index_;
};
```

```cpp

// 但测 PlaylistGenerator 不容易！

public:
    void testBadDesignHasNoSeams() {
        auto generator = std::make_unique<PlaylistGenerator>();
        // 哒！现在我们绑在了带臃肿
        // FullLibraryIndex 的
        // VideoPlaylistIndex 上，会让测试变慢。
    }

```

```cpp

        auto fake_index = std::make_shared<InMemoryVideoPlaylistIndex>();
        auto generator = std::make_unique<PlaylistGenerator>(fake_index);
```

多个构造函数，有些只为测试而用，是个信号：你的部分代码仍然会难测。左边那个例子里，`VideoPlaylistIndex` 容易测（你可以传一个测试替身 `VideoRepository`）。然而，任何用了无参构造函数的依赖对象都难测。

- 缺陷：`PlaylistGenerator` 难测，因为它用了 `VideoPlaylistIndex` 的无参构造函数，而那个被硬编码成用 `FullLibraryIndex`。你并不真想在 `PlaylistGenerator` 的测试里测 `FullLibraryIndex`，但你被迫如此。
- 缺陷：通常 `@VisibleForTesting` 注解是一种坏味道，说明这个类当初没被写成容易测试的样子。即便它能让你设置 calls 列表，那也只是绕过根本问题的一个*补丁*。

理想情况下，`PlaylistGenerator` 在构造函数里要 `VideoPlaylistIndex`，而不是直接创建自己的依赖。一旦 `PlaylistGenerator` 要了它的依赖，就没人再调无参的 `VideoPlaylistIndex` 构造函数了，我们也就自由地可以删掉它。我们通常不需要多个构造函数。

## 常见问题

**Q1**：好吧，我想我懂了。我就在构造函数里*只*做赋值，然后搞个 `init()` 方法或 `init(...)` 来干原来在构造函数里的所有活。这听起来行吗？

- **A**：我们不鼓励这个，见上面的代码示例。

**Q2**：那多个构造函数呢？能不能一个容易构造、另一个干很多活？我在测试里只用容易那个，甚至给它标上 `@VisibleForTesting`。行吗？

- **A**：我们不鼓励这个，见上面的代码示例。

**Q3**：我能把命名构造函数作为额外的、可能干活的构造函数吗？我会留一个只做字段赋值的构造函数给测试用。

- **A**：真有人问这个的话，我们会展开讲。

**Q4**：我老听你们说创建"工厂"和那些专门负责构造对象图的对象。可说真的，对象太多了，对这么简单一件事来说太过了。

- **A**：通常一个工厂对应一整个对象图，见[例子](https://web.archive.org/web/20200426113951/http://code.google.com/p/unit-test-teaching-examples/source/browse/trunk/src/di/webserver/ServerBuilder.java?r=6)。所以你不必担心每个类一个工厂导致类爆炸。工厂的职责是创建对象图、不做任何活（你该看到的尽是一堆 `new` 关键字和传来传去的引用）。对象图的职责是干活、不做对象实例化。
