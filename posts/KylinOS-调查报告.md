# KylinOS 调查报告

## 前言

提及国产操作系统，绕不开麒麟。它频繁出现在政府采购名录中，预装于政企采购的终端，名字自带"自主研发"的分量。但若追问一句"麒麟究竟是什么"，多数人答不上来，包括实际使用过它的人。

一种流传很广的说法是"麒麟即套壳 Ubuntu"。这个判断省事，却经不起推敲。麒麟的起点是 FreeBSD，并非某个 Linux 发行版的下游；它后来虽转向 Linux 内核，走的也不是 Ubuntu 的路径。至于"优麒麟"（Ubuntu Kylin），那是另一条支线，与麒麟本体不可等同。

本报告试图厘清三件事：麒麟从何而来、技术上如何构成，以及它与其它发行版的差异根子何在。其中最后一问最为关键，其答案不在补丁层面，而在管理维护的理念层面。文中事实尽量核至来源，核验不到之处明确标注，不作含糊其辞。

## 一、来历：麒麟从何而来

麒麟的起点要回溯到 2001 年。那一年，国防科技大学启动了一个操作系统项目，隶属国家 863 计划。该计划的意图很明确：让中国关键技术摆脱对外依赖。操作系统作为"卡脖子"的基础软件，自然在计划之列。系统取名"麒麟"（qilin），取自神话瑞兽，名字本身即承载着自主的期许。

这一出身决定了麒麟与绝大多数 Linux 发行版的根本不同。Ubuntu 由 Canonical 公司发起，Debian 由全球开发者社区共建，二者的诞生都是技术社区的自然生长。麒麟则不然。它从立项之初即带有国家任务的属性，目标是"自主可控"，而非单纯的"好用好卖"。这一基因，持续影响着它此后的每一次技术选择。

时间线上还有一个细节值得注意：麒麟比许多人想象的更早。2001 年立项，2006 年即被报道其服务器版本研发成功，称其为"首个达到 B2 级安全等级的 64 位操作系统"[^kylin_wiki]。那时 Ubuntu 还年幼，国内 Linux 生态也远没有今天活跃。麒麟并非后来跟风之作，它从一开始就置身局中。

## 二、技术演进：三次转身

麒麟的技术路线经历了三次明显的转向，每一次背后都是策略的重新权衡。

**第一次：立足 FreeBSD。** 早期麒麟（2001 至 2006 年前后）基于 FreeBSD 5.3。此事在 2006 年曾引发争议——一位网名为 Dancefire 的在澳中国学生匿名分析了麒麟代码，得出其与 FreeBSD 5.3 在函数名上相似度高达 99.45%、源代码相似度约 60% 的结论；开发者在 EuroBSDCon 2006 上也承认了基于 FreeBSD。表面看像是"抄袭"，但换个角度：基于一个成熟的 BSD 实现起步，远比从零编写内核现实。国防科大当时追求的是"能用的自主系统"，而非"从零证明自主"。这是一次务实的妥协。

**第二次：转向 Linux 内核。** 到 3.0 版本，麒麟更换底层，从 FreeBSD 切换至 Linux 内核。这一转变的意义超出表面：Linux 内核的生态、驱动与社区支持远超 FreeBSD，切换意味着麒麟能接入更广阔的技术资源。但需注意——它采用的是 Linux 内核本身，而非"基于 Ubuntu"或"基于 Debian"。本体麒麟并无某个发行版作为其上游，这一点恰与"套壳 Ubuntu"的说法相冲突。

**第三次：开源化，openKylin。** 2022 年 7 月，openKylin（"开放麒麟"）作为开源版本发布，由开放原子开源基金会孵化运营，采用木兰宽松许可证 v2[^openkylin_community]。商业版仍为专有，但代码以开源社区形式公开。这一步的意味，留待理念章节展开。

三次转身，串联起一条清晰的轨迹：从"自研内核"的硬扛，到"借力成熟开源"的务实，再到"开源换生态"的开放。每一次都不是技术冲动，而是策略重估。一个值得注意的观察是：麒麟从未真正"独立造内核"，它的自主始终建立在借力开源之上。这是它与"完全自研"叙事之间的真实距离。

## 三、版本谱系：不只是一条线

麒麟并非单一产品，而是一个产品家族。理清谱系，才能避免把不同事物混为一谈。

**银河麒麟（Kylin）**：本体，商业专有。按场景分为桌面版与服务器版，近年又衍生出智算操作系统（面向 AI 算力）、工业操作系统、万物智联操作系统（物联网）。主流版本由 V10 演进至 V11：V10 发布于 2020 年 8 月，V11 发布于 2025 年 8 月，号称"中国首个 6.6 内核商业版本"[^kylin_zhwiki]。这是政府采购中最常出现的一个。其运营主体为麒麟软件有限公司，系 2019 年 12 月由中标软件与天津麒麟整合而成[^kylin_zhwiki]。

**NeoKylin（中标麒麟）**：2010 年 12 月，民用"中标 Linux"与军用"银河麒麟"合并，统一品牌为"中标麒麟"（NeoKylin），其名即取"中标"与"麒麟"之合称。它一度在政企市场铺得很广，与本体麒麟属合并关系，并非完全独立的另一事物。

**Ubuntu Kylin（优麒麟）**：2013 年 4 月发布的 13.04 版本，由 Canonical（Ubuntu 母公司）与中国工信部合作推出[^kylin_wiki]。**关键在于：它基于 Ubuntu，是独立合作项目，不等于麒麟本体。** 许多人对"麒麟即套壳 Ubuntu"的印象，正源于此。但优麒麟只是麒麟家族的一条支线，本体麒麟并不基于 Ubuntu。

**openKylin（开放麒麟）**：2022 年开源的社区版，前文已述。它是商业银河麒麟的开源对应面。

四者并置，麒麟家族的"双轨"格局便清晰起来：商业专有（银河麒麟）与社区开源（openKylin）并行，中间夹着合作支线（NeoKylin、Ubuntu Kylin）。这一布局与 Fedora→RHEL 的单向上下游不同：Fedora 是 RHEL 的上游试验田，关系线性；麒麟这边，商业版与社区版更近于并行的两条线，互非试验田。这种双轨本身即是一种理念选择，留待第十一章展开。

## 四、技术架构：保守与可控

就架构而言，麒麟并无多少花哨之处。Linux 内核，UKUI 桌面，apt/dpkg 系包管理。支持平台挺广泛：amd64、arm64、alpha、loongarch64、mips64el——后四者分别对应申威、鲲鹏/飞腾、龙芯、MIPS 系国产 CPU，这正是信创全栈自主在架构层的体现。

值得多说一句的是桌面环境 **UKUI**。其全称为 Ultimate Kylin User Interface，以 Qt 编写，但身世略有曲折：它是 MATE 桌面的一个 fork（MATE 又源自 GNOME 2），历史上曾从 GTK 迁移至 Qt。稳定版 4.0 于 2023 年发布[^ukui_wiki]。视觉上，UKUI 接近 Windows 7，资源占用不高，对老旧设备及由 Windows 迁入的用户较为友好。

此处有一个不甚显眼却挺关键的选择：麒麟自研了 UKUI，而非直接采用 GNOME 或 KDE。直接沿用现成桌面固然省事，但在"自主可控"的逻辑下，连桌面这一层都得攥在自己手里。UKUI 的存在，与其说是技术上更优，不如说是"可控优先"理念在桌面层的投影。技术选择服务于理念，这里先埋一笔，后文收束。

包管理采用 apt/dpkg，表明 openKylin/银河麒麟隶属 Debian 系包生态。这一点并不矛盾：内核为 Linux（非基于某发行版），但包管理借用了 Debian 体系的成熟工具。自主不等于事必躬亲，工具可用则用，这是另一层务实。

## 五、安全机制与信创

前文提及，麒麟于 2006 年即被报称达到"B2 级安全等级"。B2 是 TCSEC（美国可信计算机系统评估准则）中的结构化保护级，对内核安全机制要求挺高。这一等级标签是麒麟早期便立住的人设，它从一开始就把"安全"作为核心卖点。

但麒麟的"安全"与国际主流发行版的安全并非同一路数。Ubuntu、Debian 讲求的安全，是漏洞快速响应、CVE 修复、最小权限；麒麟讲求的安全，则更多绑定于"等级保护"与"信创合规"：系统能否通过等保测评、能否进入采购名录、是否符合自主可控的要求。二者并非一回事，前者是攻防视角的工程安全，后者是合规视角的制度安全。麒麟偏于后者。

理解了这一点，许多事便顺理成章。麒麟为何大力适配国产 CPU？因为信创采购要求软硬件全栈自主，CPU 得是国产，系统也得能跑在国产 CPU 上。为何强调"自主可控"？因为合规审计要审视代码可控性与供应链可控性。安全在此并非抽象的"更难被攻破"，而是具体到"能否进入某个采购目录"。理念决定了优先级：合规优先，工程安全反成副产品。

至 V11，这套安全有了具体的技术载体。官网称服务器版 V11 以 KSAF 安全框架为核心，"全面支持国家商用密码标准"，即国密算法（SM 系列）。国密是信创安全的硬指标之一，以国产密码算法替代 RSA/AES 等国际算法，是自主可控在加密层的落地。配套的还有系统热补丁：无需重启即可动态修复内核及关键组件的安全漏洞。热补丁技术本身并不新鲜（上游 Linux 早有），但麒麟将其做成产品级特性并重点宣传，贴合的是政企场景"不能停机"的诉求——合规安全不止要通过测评，还得保证业务不中断。这些特性单拎出来都不稀奇，但组合在一起服务于"合规、可控、不停"的目标，才是麒麟安全的真实样貌。


## 六、生态与应用

论及生态，麒麟的数据挺可观。据维基记载，截至 2019 年，麒麟与 NeoKylin 合计占据政府领域约 90% 份额；NeoKylin 兼容超 4000 个软硬件产品；2015 年戴尔报告其在华销售的 42% 个人电脑预装 NeoKylin[^kylin_wiki]。英文维基另提及天河一号、天河二号超算采用 64 位麒麟 Linux（中文维基对此未作记载，存此备考）[^kylin_wiki]。Kylin V10 发布时宣称兼容上万个软硬件产品，廖湘科也宣布采用麒麟的党政企业已逾万家[^kylin_wiki]。

但这里要分清两件事：**份额与活力。** 政府领域 90% 的份额，是政策驱动的必然结果——信创采购明确要求采用国产系统，份额高是制度使然，并不代表开发者社区活跃。戴尔预装 NeoKylin，同样是配合政企采购需求的商业动作。换言之，麒麟的生态是"政策与适配"喂养而成，并非社区自然生长。这与 Ubuntu、Debian 那种依托全球开发者贡献、应用自然涌现的生态，是两种截然不同的增长模式。

哪种更好，见仁见智。政策驱动的好处是落地快、覆盖广，短板是脱离采购场景后民间活力有限。一个直观的对比：在 Ubuntu 上可轻松找到某个冷门库的打包，在麒麟上则未必，因为少有人有动力去打包采购清单之外的东西。生态的形态，随理念而定。

## 七、信创软件生态：从无到有到壮大

单看麒麟一系统，生态是单薄的。但麒麟并非孤立存在，它生长于一个更大的格局之中——信创（信息技术应用创新）产业。这一格局的体量与形态，决定了麒麟能借多少力。本章要跳出来看全局。

信创并非凭空而生。它起步于 2014 年的金融领域，当年银监会提出"去 IOE 化"，即以国产替代 IBM 小型机、Oracle 数据库、EMC 存储等外国核心 IT。2016 年信创工委会成立，开始制定标准、组织培训、统筹推进。真正加速是在外部压力到来之后——美国的制裁使"自主可控"由口号变为刚需，信创被视作关键技术独立的抓手[^xinchuang_wiki]。这就是大背景：政策定调、外部倒逼、自上而下推进。

信创格局之中不止操作系统。它分为四大块：基础设施（CPU、芯片、服务器）、基础软件（操作系统、数据库、中间件、云平台）、应用软件（办公、通用、行业软件）、信息安全[^xinchuang_wiki]。麒麟仅是基础软件中的一条。正因格局庞大，每一块都有国产厂商在填补：CPU 有龙芯、兆芯、华为鲲鹏、飞腾；数据库有达梦、人大金仓；办公软件有 WPS、永中；安全有奇安信、360。这些厂商相互适配、相互背书，构成一个"全栈国产"的闭环。麒麟的价值，很大程度上源自它是这一闭环中操作系统那一环。脱离龙芯飞腾的 CPU、WPS 的办公套件，麒麟单拎出来撑不起一个办公场景；但置于信创栈中，它便齐备了。

推广是分阶段的：先于政府、军队、国企试点，再扩展至金融、电信、电力、交通等关键行业，最后才面向消费市场。这一节奏解释了为何麒麟在政企领域份额高、在普通消费者中却近乎无存在感：它尚未真正触及消费市场，前两阶段尚未走完。规模上，政府信创合同从 2011—2018 年累计约 2 亿，至 2020 年一年即跃升至 279 亿，有预测 2025 年市场规模破 2 万亿。数字增长虽快，但大头由政策采购支撑，而非市场自发选择。

生态壮大的真实面貌，可概括为"适配"二字。麒麟设有专门的生态适配中心（官网为 eco.kylinos.cn），其职责便是将各类国产软硬件与系统配齐、测通、颁发证书。WPS 出新版要和麒麟适配，龙芯出新 CPU 也要和麒麟适配，反过来一样。这种适配驱动的生态，与开源社区"有人写即有人用"的自然增长不同，它是"政策要求必须配齐，遂组织力量配齐"。结果是：关键场景能够跑通（办公、安全、政企核心业务），但长尾应用与新潮工具的覆盖，与国际主流发行版相去甚远。

故而，信创生态确已从无到有壮大，这一点并没有水分。但其壮大的形态是"政策驱动 + 适配填充"，而非"开发者涌入 + 应用涌现"。理解了这一点，前文所述麒麟生态之薄便不矛盾：整个信创生态都偏此路，麒麟只是其中一个切面。它不孤单，但也不轻快。

## 八、磐石架构：不可变理念的麒麟版

近年来，不可变（immutable）操作系统是一明显趋势。Fedora Silverblue、SteamOS、NixOS 都将系统目录设为只读，更新走原子化，故障可回滚，其长处在于系统状态稳定、不易损坏。麒麟在此方向上也有所跟进，只是用了自己的命名——桌面版 V11 官网提及一项名为"磐石架构"的设计。

据官网所述，磐石架构配合"开明包格式"，实现"核心基础系统与应用的隔离式创新设计"，从底层架构层面提升系统更新的稳定性与可靠性，增强软硬件兼容性。换言之：将系统核心与应用隔离，应用归应用、系统归系统，互不污染。这一思路与 Silverblue 的只读根加容器化应用一脉相承，只是麒麟未照搬其术语，另起了一套命名。

这一选择挺有麒麟风格。国际发行版推行不可变，多是在现有发行版之上套加一层（Silverblue 即 Fedora 的变体）；麒麟则将架构写入自家产品名与卖点，强调"此为自研架构"。技术内核相通（系统应用隔离、更新稳定），但包装与归属感为己所有。这与前文 UKUI 自研的逻辑一致：可借的理念尽可借，但名称与主导权得攥在自己手里。可控优先，连架构命名也不例外。

至于磐石架构具体如何实现隔离、是否为完整只读根、回滚机制做到何种程度，官网只止于营销话术层面的描述，没有技术白皮书级的细节公开。所以这里点到"方向与理念"为止，不假装比官方说得更细。

## 九、应用与包管理：把应用关进笼子

应用这一层，麒麟同时并存几套体系。

传统的是 deb/rpm 包，apt/dpkg 那一套，与 Debian 系发行版共用生态。这是基础，但传统包管理有一老问题：依赖冲突、系统级污染，装多了易致系统紊乱。

于是麒麟转向沙箱化。一条线是社区的**玲珑（linglong）**——国产系统打造的一套包管理格式，主打沙箱化，应用运行于隔离环境，不污染系统，权限可控，思路与 Flatpak、Snap 同类，都属于"将应用关入笼中"。另一条线是商业版 V11 自有的**开明包格式**——官网将其与磐石架构并提，做"核心基础系统与应用的隔离"，同样是不让应用擅动系统核心。

两条线一开源一商业，但理念同源：传统包是"信任应用、给它系统权限"，沙箱包是"不信任应用、只给最小边界"。这种"可控分发"的取向，与前文安全、桌面、架构诸层的逻辑一脉相承——麒麟在每一技术选择上，都倾向于"把东西攥在自己手里、限定边界"。开明包格式这一名称本身也透出意味：开明，但仍包裹、仍管束。

应用生态另有一块是安卓兼容。麒麟 V10 发布时即提及支持安卓生态，可在桌面运行安卓应用。这也是国产系统弥补 Windows 软件缺口的常见手段：原生 Linux 应用既然不足，便将安卓应用生态接入填空。V11 进一步将 AI 应用生态纳入：官网称系统集成主流 AI 算力平台与框架，支持 DeepSeek（并附适配测试证书），兼容 CUDA、oneAPI，配套 AI 助手 kyCopilot 与调优工具 Extuner。将当下最热的 AI 栈也纳入适配，说明麒麟的生态补缺不止于"能跑办公软件"，也着眼于"能跑大模型"。


## 十、内核源码：和主线 Linux 差在哪

本节往下挖一层，看麒麟的内核究竟与主线 Linux 差在何处。主角是商业版银河麒麟 V10/V11，但有一绕不开的现实：商业版内核专有，补丁不公开，无从直接探究。能看见真东西的，是其开源对应面 openKylin 的内核仓库（`gitee.com/openkylin/kernel`，已迁至 GitCode 维护）。这里要讲清楚：**openKylin 内核不等于商业版内核，但二者同源，开源仓库可作为商业版内核取向的旁证**，而非直接证据。

**基线：** openKylin 内核主要基于 Linux 6.6 LTS，同时维护 6.1、5.15，并有 6.12、7.0 的开发线。许可证为标准 GPL-2.0。也就是说其根仍在上游 Linux，并未另起炉灶重造内核。这一点另有商业版侧的佐证：V11 自称"中国首个 6.6 内核商业版本"，与开源仓库的 6.6 基线相互印证，说明商业版与开源版在内核基线上确系同源。

**与主线 Linux 的差别，可从仓库实据看出：**

- **上游所无的定制文件**：`Kconfig.kylin`、`Makefile.kylin`、`MAINTAINERS-DIR`。这几个文件是麒麟专属的内核配置与构建规则，主线中并没有。`Kconfig.kylin` 大概率定义了麒麟专属的 CONFIG 选项。
- **新增 CONFIG 选项**：可见向内核中加入主线所无的配置项，例如有 PR 标题为 `Add CONFIG_DRM_MWV207=m`，为某显示驱动开启编译开关（疑与国产 GPU 相关）。主线不会为某一特定硬件专门增设此类开关，麒麟则会。
- **版本号方案**：标签形如 `dev/6.6.0-15.0ok10`、`build/6.6.0-23.0ok2`，其中 `ok` 后缀即 openKylin 自有的版本标记。
- **Debian 风格打包**：仓库中含 `debian/`、`debian.master/` 目录，表明内核打包沿用 Debian 体系。
- **分支布局**：约 19 个分支中，有针对国产 CPU 的适配分支，如 `linux-zhaoxin-5.15`（兆芯）、`sandbox/intel/`（Intel）。另有一组以河流命名的发行版分支——`openkylin/huanghe`（黄河）、`openkylin/nile`（尼罗）、`openkylin/yangtze`（长江），对应不同的 openKylin 发行版本。

**讲不清的部分，也要说明：** 上述为结构性差异，可从仓库布局与文件名直接看出。但具体补丁集——麒麟对上游 6.6 究竟改了多少行、改了哪些驱动、飞腾与鲲鹏的调优具体如何——需要 `git diff upstream` 才能详列，而 Gitee 上的仓库已转为只读归档，无法运行 git 命令对比。所以本节点到结构性差异为止，不逐条列举补丁，更不编造"麒麟为飞腾增设了某项调优"这类未核验的细节。

即便如此，本节的结论已足够清晰：麒麟内核（以 openKylin 仓库为旁证）并非主线 Linux 的简单复刻，它维护自己的 LTS 基线、增设自己的 CONFIG、走自己的打包、为国产 CPU 留有专门分支。这些动作拼合起来，即是"自主"在内核层的具体体现——不只是跟随上游，而是有自己的维护节奏与定制层。商业版 V10/V11 的专有补丁藏于水面之下（桌面版 V11 官网提及"基于全新内核开发"，具体改了什么未公开），但从开源这一侧，已可推断麒麟在内核层确实做了自己的事。

## 十一、管理维护理念：和其他发行版的分野

前文十章埋了不少伏笔：国家意志的诞生、三次转身的策略、双轨格局、UKUI 的自研、合规优先的安全、政策驱动的生态、内核的自有基线。现将它们收束起来，正面回答那个核心问题——麒麟与其他发行版，究竟差在何处？

先说定位。麒麟（其开源面 openKylin 尤为强调这一点）以"根社区"自居——不依赖别的发行版作上游，自身即技术栈之根。Ubuntu 背后是 Debian，Fedora 自身即上游但服务于 RHEL，二者都有明确的上下游归属。麒麟要的是"我是一条根"，而非"我是某条根的下游"。这一念头决定了它维护内核基线、做自己的包格式、不肯直接采用 GNOME——每一步都在强化"我是独立一极"的身份感。

治理上也能看出差别。openKylin 由开放原子开源基金会孵化，技术委员会下设 SIG 组，贡献者要签 CLA，许可证为木兰宽松许可证 v2。这套机制组织色彩浓重——基金会托管、有架构、贡献得走协议。Debian 是全球开发者散漫自治、归 SPI 管辖；Fedora 背靠 Red Hat，社区与商业之间上下游分明。麒麟的治理更"有主"，不像 Debian 那样松散。

但最根本的差异在驱动力上。Ubuntu、Fedora 维护系统，图的是技术先进与社区共识：出新功能、修 CVE、跟上游节奏，目标是"做个好用的发行版"。麒麟维护系统，图的是自主可控与信创合规：能否适配国产 CPU、能否过等保、能否进采购名录，目标是"做个合规可控的国产系统"。一个是工程逻辑，一个是制度逻辑。同一件维护内核之事，动机不同，优先级便不同，长出来的系统自然相异。这条差异是底层的，前三条多少都由它派生。

还有双轨这一层结构。商业银河麒麟专有与社区 openKylin 开源，两条线并行。这与 Fedora→RHEL 的"社区上游、商业下游"恰相反——Fedora 是 RHEL 的试验田，关系线性；麒麟的商业版与社区版更近于并列，各自服务不同场景。这种双轨背后，是"既要商业落地、又要开源换生态"的平衡。

将这几点并置，麒麟的差异便清晰了：它并非技术水准上的差异（内核仍是 Linux、包仍是 deb），而是**为何维护、为谁维护、如何组织维护**的差异。麒麟那些显得"怪"之处——自研桌面、合规安全、政策生态、根社区定位——都可以从这套理念中推出。理念是因，技术选择是果。

## 十二、客观评价：理念是有代价的

讲了这么多麒麟的理念，容易显得在称颂。得补一刀：这套理念是有代价的，且代价不轻。

最直接的代价是生态单薄。当生态依靠政策与适配驱动、而非社区自发，脱离采购场景的活力便有限。一个普通开发者想为麒麟贡献某个冷门应用，门槛与动力都高于为 Ubuntu 贡献。社区薄，应用便少，用户体验的天花板便低。这是理念换来的现实，无法两全。

双轨也分散精力。商业版与社区版并行，听来两全其美，实则两边都需要维护和投入。openKylin 社区的活跃度，与真正的根社区（Debian、Fedora）相比尚有距离——"根社区"的定位与社区的实际体量之间，落差摆在那里。

更隐蔽的代价是创新慢半拍。什么事都攥在自己手里，UKUI 自研、开明包格式自研，长处是可控，短处是难借力。国际社区搞出的新事物，麒麟得等自己适配、自己可控了才能采用，节奏天然偏慢。有时并非做不到，而是走完"可控"这道流程，时机已然错过。

与同类国产系统相比，麒麟与统信 UOS、Deepin 是直接竞品。Deepin 走的是"做好桌面体验、攒社区口碑"的路子，UOS 是 Deepin 的商业版。三者理念有重叠（都求自主可控），但侧重不同：麒麟偏合规与政企，Deepin 偏桌面体验与用户好感，UOS 居中。孰更好用？视场景而定。政企采购多选麒麟，个人折腾桌面或更青睐 Deepin。

最后说句实在话：麒麟不是"最好的 Linux 发行版"，它也未打算做这个。它要的是"合规、可控、能落地"的国产系统，在它要解决的场景里，它做到了。理解麒麟，得先理解它并非为与 Ubuntu 比好用而生，而是为在特定约束下把"自主可控"做成可交付的产品。理念决定了它的样貌，也划定了它的边界。

调查至此。一句话总结：麒麟是为国家意志与信创合规而生的 Linux，技术选择处处透着"可控优先"，这使它达成了自主可控的目标，也背负着生态与活力的代价。
