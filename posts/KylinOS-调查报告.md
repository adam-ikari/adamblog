# KylinOS 调查报告

## 前言

提起国产操作系统，绕不开麒麟。它出现在政府采购名录里，预装在政企采购的 PC 上，名字带着一股"自主研发"的郑重感。但真要问一句"麒麟到底是个什么东西"，多数人答不上来——包括用过它的人。

常见的说法是"麒麟就是套壳 Ubuntu"。这个判断省事，但经不起推敲。麒麟的起点是 FreeBSD，不是 Linux 发行版的下游；它后来转用了 Linux 内核，但走的也不是 Ubuntu 那条路。至于"优麒麟"（Ubuntu Kylin），那是另一个故事，跟本体麒麟不能画等号。

这篇报告想搞清楚的，是三件事：麒麟从哪来、技术上怎么构成的，以及它和其他发行版的差异，根子到底在哪（这最后一问最重要）。答案不在补丁里，而在管理维护的理念里。事实部分尽量核到来源，核不到的地方会标明，不装懂。

## 一、来历：麒麟从哪来

麒麟的故事要从 2001 年说起。那一年，国防科技大学启动了一个操作系统项目，挂在国家 863 计划名下。863 计划的野心写在明面上：让中国关键技术摆脱对外依赖。操作系统这种"卡脖子"的基础软件，自然在计划之内。系统取名"麒麟"（qilin），神话里的瑞兽，名字里就带着自主的期许。

这个出身决定了麒麟和绝大多数 Linux 发行版不一样。Ubuntu 是 Canonical 公司攒出来的，Debian 是全球开发者社区攒出来的，它们的诞生是技术社区的自然生长。麒麟不是。它从第一天起就带着国家任务的属性，目标是"自主可控"，不是"好用好卖"。这个基因，会一路影响它后来的每一个选择。

顺带说一句时间线上的细节：麒麟比很多人想象的要早。2001 年立项，2006 年军方就报道了服务器版本研发成功，称其为"首个达到 B2 级安全等级的 64 位操作系统"。那时候 Ubuntu 才两岁，国内的 Linux 圈子还没今天这么热闹。麒麟不是后来跟风造的，它一开始就在牌桌上了。

## 二、技术演进：三次转身

麒麟的技术路线，走过三次明显的转身。每一次转身，背后都是一种策略的调整。

**第一次：站在 FreeBSD 肩上。** 早期的麒麟（2001 到 2006 年前后）基于 FreeBSD 5.3。这事在 2006 年闹过一阵——一个在澳大利亚的中国学生匿名分析了麒麟代码，发现它和 FreeBSD 5.3 的相似度高达 99.45%。开发者在 EuroBSDCon 2006 上也承认了基于 FreeBSD。听起来像是"抄"，但换个角度：站在一个成熟的 BSD 实现上起步，比从零写一个内核现实得多。国防科大当时要的是"能用的自主系统"，不是"从零证明自己"。这是一次务实的妥协。

**第二次：转向 Linux 内核。** 到了 3.0 版本，麒麟换了底，从 FreeBSD 切到 Linux 内核。这一换的意义，比表面看起来大。Linux 内核的生态、驱动、社区支持都远超 FreeBSD，切过去意味着麒麟能搭上更宽的车。但要注意——它用的是 Linux 内核，不是"基于 Ubuntu"或"基于 Debian"。本体麒麟没有哪个发行版做它的上游，这一点和后来流传的"套壳 Ubuntu"说法是冲突的。

**第三次：开源化，openKylin。** 2022 年 7 月，openKylin（"开放麒麟"）作为开源版本发布，由开放原子开源基金会孵化运营，许可证是木兰宽松许可证 v2。商业版仍然专有，但代码以开源社区的形式公开了。这一步的意味，后面讲理念时再说。

三次转身，凑起来是一条清晰的轨迹：从"自己造一个内核"的硬扛，到"借力成熟开源"的务实，再到"开源换生态"的开放。每一次都不是技术冲动，而是策略重估。一个有意思的观察：麒麟从没真正"独立造内核"过——它的自主，始终是建立在借力开源之上的自主。这是它和"完全自研"叙事之间的真实距离。

## 三、版本谱系：不只是一条线

麒麟不是单一产品，是一大家子。理清谱系，才不会把不同东西混为一谈。

**银河麒麟（Kylin）**：本体，商业专有。按场景分桌面版、服务器版，近年还衍生出智算操作系统（面向 AI 算力）、工业操作系统、万物智联操作系统（物联网）。主流版本从 V10 到 V11（V10 发布于 2020 年 8 月）。这是政府采购里最常出现的那一个。

**NeoKylin（中标麒麟）**：2010 年国防科大与中标软件合作的产物，名字就是"中标"软件 + "麒麟"的合称。中标软件本身做 NeoShine Linux，两边合流弄出 NeoKylin，一度在政企市场铺得挺开。它和本体麒麟是合作关系，不是一回事。

**Ubuntu Kylin（优麒麟）**：2013 年 4 月发布的 13.04 版本，是 Canonical（Ubuntu 母公司）和中国工信部合作的产物。**关键点：它基于 Ubuntu，是独立合作项目，不等于麒麟本体。** 很多人把"麒麟=套壳 Ubuntu"的印象，就是从优麒麟来的。但优麒麟是麒麟家族里的一条支线，本体麒麟并不基于 Ubuntu。

**openKylin（开放麒麟）**：2022 年开源的社区版，前文提过。它是商业银河麒麟的开源对应面。

把这四者摆一起，麒麟家族的"双轨"格局就清楚了：商业专有（银河麒麟）+ 社区开源（openKylin）两条腿走路，中间夹着合作支线（NeoKylin、Ubuntu Kylin）。这个布局和 Fedora→RHEL 的单向上下游不一样——Fedora 是 RHEL 的上游试验田，关系是线性的；麒麟这边，商业版和社区版更像是并行的两条线，谁也不是谁的试验田。这种双轨，本身就是一种理念选择，第十一章再展开。

## 四、技术架构：保守与可控

讲架构，麒麟没什么花哨的。Linux 内核，UKUI 桌面，apt/dpkg 系包管理，支持 x86-64 和 ARM。信创场景下还适配龙芯、飞腾、鲲鹏等国产 CPU（这部分以官方最新适配名录为准）。

值得多说的是桌面环境 **UKUI**。它的全称是 Ultimate Kylin User Interface，用 Qt 写的，但身世有点曲折——它是 MATE 桌面的一个 fork（MATE 又源自 GNOME 2），历史上从 GTK 迁到了 Qt。稳定版 4.0 在 2023 年发布。视觉上，UKUI 长得像 Windows 7，资源消耗不大，对老机器和从 Windows 切过来的用户都友好。

这里有个不太显眼但挺关键的选择：麒麟自研了 UKUI，而不是直接用 GNOME 或 KDE。直接用现成桌面省事，但"自主可控"的逻辑下，连桌面这一层都得握在自己手里。UKUI 的存在，与其说是技术上更优，不如说是"可控优先"理念在桌面层的投影。技术选择服务于理念，这一节就埋一个点，后面收。

包管理用 apt/dpkg，说明 openKylin/银河麒麟属于 Debian 系的包生态。这一点倒不矛盾：内核是 Linux（非基于某发行版），但包管理借了 Debian 体系的成熟工具。自主不等于什么都自己造，工具能用就用，这是另一层务实。

## 五、安全机制与信创

前面提过，麒麟 2006 年就被报称达到"B2 级安全等级"。B2 是 TCSEC（美国可信计算机系统评估准则）里的结构化保护级，对内核安全机制要求不低。这个等级标签是麒麟早期就立住的人设——它从一开始就把"安全"当作核心卖点。

但麒麟的"安全"和国际主流发行版的安全，不是一个路数。Ubuntu、Debian 讲安全，讲的是漏洞快速响应、CVE 修复、最小权限。麒麟讲安全，更多绑着"等级保护"和"信创合规"：系统能不能过等保测评、能不能进采购名录、符不符合自主可控的要求。这俩不是一回事，一个是攻防视角的工程安全，一个是合规视角的制度安全。麒麟偏后者。

理解了这点，很多事就顺了。为什么麒麟花大力气适配国产 CPU？因为信创采购要求软硬件全栈自主，CPU 得是国产的，系统得能跑在国产 CPU 上。为什么强调"自主可控"？因为合规审计要看代码可控性、供应链可控性。安全在这里不是抽象的"更难被黑"，而是具体到"能不能进某个采购目录"。理念决定了优先级：合规优先，工程安全是副产品。

到 V11，这套安全有了具体的技术载体。官网讲服务器版 V11 以 KSAF 安全框架为核心，"全面支持国家商用密码标准"——也就是国密算法（SM 系列）。国密是信创安全的硬指标之一，能用国产密码算法替代 RSA/AES 这些国际算法，是自主可控在加密层的落地。配套的还有系统热补丁：不用重启就能动态修内核和关键组件的安全漏洞。热补丁这技术不算新鲜（上游 Linux 早有），但麒麟把它做成产品级特性并重点宣传，贴合的是政企场景"不能停机"的诉求——合规安全不止要过测评，还得保证业务不中断。这些特性单拎出来都不稀奇，凑在一起服务于"合规、可控、不停"的目标，才是麒麟安全的样子。


## 六、生态与应用

讲生态，麒麟的数字挺漂亮。维基记载，截至 2019 年，麒麟和 NeoKylin 合计占政府领域约 90% 份额；NeoKylin 兼容超 4000 个软硬件产品；2015 年戴尔报告其在华销售的 42% 个人电脑预装 NeoKylin。天河一号、天河二号超算用的也是 64 位麒麟 Linux。Kylin V10 发布时宣称兼容上万个软硬件产品。

但这里要分清两件事：**份额和活力。** 政府领域 90% 的份额，是政策驱动的结果，信创采购明确要求用国产系统，份额高是必然的，不代表开发者社区活跃。戴尔预装 NeoKylin，也是配合政企采购需求的商业动作。换句话说，麒麟的生态是"政策和适配"喂出来的，不是社区自然生长出来的。这和 Ubuntu、Debian 那种靠全球开发者贡献、应用自然涌现的生态，是两种完全不同的增长模式。

哪种好？见仁见智。政策驱动的好处是落地快、覆盖广，坏处是离开采购场景，民间活力有限。一个直观的对比：你在 Ubuntu 上能轻松找到某个冷门库的打包，在麒麟上未必，因为没人有动力去打包一个采购清单外的玩意。生态的形态，跟着理念走。

## 七、信创软件生态：从无到有到壮大

单看麒麟一个系统，生态是单薄的。但麒麟不是孤立存在的，它长在一个更大的盘子里：信创（信息技术应用创新）产业。这个盘子有多大、长得怎么样，决定了麒麟能借多少力。这章得跳出来看全局。

信创不是凭空冒出来的。它起步于 2014 年的金融领域，那年银监会提出"去 IOE 化"——把 IBM 小型机、Oracle 数据库、EMC 存储这些外国核心 IT 换掉。2016 年信创工委会成立，开始定标准、搞培训、组织推进。真正加速是外部压力来了之后，美国的制裁让"自主可控"从口号变成刚需，信创被当成关键技术独立的抓手。这是大背景：政策定调、外部倒逼、自上而下推。

信创盘子里不止操作系统。它分四大块：基础设施（CPU、芯片、服务器）、基础软件（操作系统、数据库、中间件、云平台）、应用软件（办公、通用、行业软件）、信息安全。麒麟只是基础软件里的一条。但正因为盘子大，每块都有国产玩家在填：CPU 有龙芯、兆芯、华为鲲鹏、飞腾；数据库有达梦、人大金仓；办公软件有 WPS、永中；安全有奇安信、360。这些玩家互相适配、互相背书，凑成一个"全栈国产"的闭环。麒麟的价值，很大程度来自它是这个闭环里操作系统那一环。离开了龙芯飞腾的 CPU、WPS 的办公套件，麒麟单拎出来撑不起一个办公场景；但放进信创栈里，它就齐了。

推广是分阶段的：先政府、军队、国企试点，再扩到金融、电信、电力、交通等关键行业，最后才面向消费市场。这节奏解释了为什么麒麟在政企领域份额高、在普通消费者里几乎没存在感：它压根还没真正打到消费市场，前两阶段还没走完。规模上，政府信创合同从 2011-2018 年累计约 2 亿，到 2020 年一年就跳到 279 亿，有预测 2025 年市场规模破 2 万亿。数字涨得快，但大头是政策采购撑的，不是市场自发选的。

生态壮大的真实样子，是"适配"二字。麒麟有专门的生态适配中心（官网给了 eco.kylinos.cn），干的就是把各种国产软硬件跟系统配齐、测通、发证书。WPS 出个新版得跟麒麟适配，龙芯出个新 CPU 得跟麒麟适配，反过来也一样。这种适配驱动的生态，和开源社区"有人写就有人用"的自然增长不一样，它是"政策要求必须配齐，那就组织力量配齐"。结果是：关键场景能跑通（办公、安全、政企核心业务），但长尾应用、新潮工具的覆盖，和国际主流发行版差得远。

所以信创生态确实从无到有壮大了，这点没有水分。但壮大的形态是"政策驱动 + 适配填充"，不是"开发者涌入 + 应用涌现"。理解了这点，前面讲麒麟生态薄就不矛盾了：整个信创生态都偏这一路，麒麟只是其中一个切面。它不孤单，但也不轻快。

## 八、磐石架构：不可变理念的麒麟版

最近几年，不可变（immutable）操作系统是个明显趋势。Fedora Silverblue、SteamOS、NixOS 都把系统目录设成只读，更新走原子化，坏了能回滚，好处是系统状态稳定、不易被搞坏。麒麟在这个方向上跟进了，不过用的是自己的名字——桌面版 V11 官网提到一个叫"磐石架构"的东西。

按官网说法，磐石架构配合"开明包格式"，做的是"核心基础系统与应用的隔离式创新设计"，从底层架构层面提升系统更新的稳定性和可靠性，增强软硬件兼容性。翻译成大白话：把系统核心和应用隔离开，应用归应用、系统归系统，互不污染。这思路和 Silverblue 的只读根 + 容器化应用是一脉相承的，只是麒麟没照搬那套术语，自己起了一套名字。

这个选择挺麒麟。国际发行版搞不可变，多是在现有发行版上套一层（Silverblue 是 Fedora 的变体）；麒麟是把架构写进自己的产品名和卖点里，强调"这是我们自研的架构"。技术内核是相通的（系统应用隔离、更新稳定），但包装和归属感是自己的。这和前面 UKUI 自研的逻辑一样：能借的理念要借，但名字和主导权得握在手里。可控优先，连架构命名都不例外。

至于磐石架构具体怎么实现隔离、是不是完整只读根、回滚机制做到什么程度——官网是营销话术层面的描述，没有技术白皮书级的细节公开。所以这里点到"方向和理念"为止，不假装比官方说得更细。

## 九、应用与包管理：把应用关进笼子

应用这一层，麒麟同时存在几套体系。

传统的是 deb/rpm 包，apt/dpkg 那一套，和 Debian 系发行版共用生态。这是基础，但传统包管理有个老问题：依赖冲突、系统级污染，装多了容易把系统搞乱。

于是麒麟往沙箱化方向走。一条线是社区的**玲珑（linglong）**——国产系统搞的一套包管理格式，主打沙箱化，应用跑在隔离环境里，不污染系统，权限可控，思路和 Flatpak、Snap 是一类的，都是"把应用关进笼子"。另一条线是商业版 V11 自己的**开明包格式**——官网把它和磐石架构并提，做"核心基础系统与应用的隔离"，同样是不让应用乱动系统核心。

两条线一个开源一个商业，但理念同源：传统包是"信任应用、给它系统权限"，沙箱包是"不信任应用、给它最小边界"。这种"可控分发"的取向，和前面安全、桌面、架构那几层的逻辑一脉相承——麒麟在每一个技术选择上，都倾向于"把东西握在手里、限定边界"。开明包格式这个名字本身也透着意思：开明，但还是包着、管着。

应用生态还有一块是安卓兼容。麒麟 V10 发布时就提到支持安卓生态，能在桌面跑安卓应用。这也是国产系统补 Windows 软件缺口的常见手段：既然原生 Linux 应用不够多，就把安卓应用生态接进来填空。V11 进一步把 AI 应用生态也接了进来：官网说系统集成主流 AI 算力平台和框架，支持 DeepSeek（还列了适配测试证书），兼容 CUDA、oneAPI，配套 AI 助手 kyCopilot 和调优工具 Extuner。把当下最热的 AI 栈也纳入适配，说明麒麟的生态补缺不止于"能跑办公软件"，还盯着"能跑大模型"。


## 十、内核源码：和主线 Linux 差在哪

这一节往下挖一层，看麒麟的内核到底和主线 Linux 差在哪。主角是商业版银河麒麟 V10/V11——但有个绕不开的现实：商业版内核专有，补丁不公开，直接挖不动。能看见真东西的，是它的开源对应面 openKylin 的内核仓库（`gitee.com/openkylin/kernel`，已迁到 GitCode 维护）。这里要讲清楚：**openKylin 内核不等于商业版内核，但二者同源，开源仓库可以作为商业版内核取向的旁证**，不是直接证据。

**基线：** openKylin 内核主要基于 Linux 6.6 LTS，同时维护 6.1、5.15，还有 6.12、7.0 的开发线。许可证是标准的 GPL-2.0。也就是说，它的根还是上游 Linux，没有另起炉灶造内核。

**和主线 Linux 的差别，能从仓库实据看到的：**

- **上游不存在的定制文件**：`Kconfig.kylin`、`Makefile.kylin`、`MAINTAINERS-DIR`。这几个文件是麒麟专属的内核配置和构建规则，主线里没有。`Kconfig.kylin` 大概率定义了麒麟专属的 CONFIG 选项。
- **新增 CONFIG 选项**：能看到往内核里加主线没有的配置项，比如有 PR 标题是 `Add CONFIG_DRM_MWV207=m`，给某个显示驱动开了编译开关（疑似国产 GPU 相关）。主线不会为某个特定硬件专门加这种开关，麒麟会。
- **版本号方案**：标签长成 `dev/6.6.0-15.0ok10`、`build/6.6.0-23.0ok2` 这样，那个 `ok` 后缀是 openKylin 自己的版本标记。
- **Debian 风格打包**：仓库里有 `debian/`、`debian.master/` 目录，说明内核打包走 Debian 体系。
- **分支布局**：约 19 个分支里，有针对国产 CPU 的适配分支，比如 `linux-zhaoxin-5.15`（兆芯）、`sandbox/intel/`（Intel）。还有一组以河流命名的发行版分支——`openkylin/huanghe`（黄河）、`openkylin/nile`（尼罗）、`openkylin/yangtze`（长江），对应不同 openKylin 发行版本。

**讲不清的部分，也要说清楚：** 上面是结构性差异，能从仓库布局和文件名直接看到。但具体补丁集，比如麒麟对上游 6.6 到底改了多少行、改了哪些驱动、飞腾和鲲鹏的调优具体怎么做，这需要 `git diff upstream` 才能详列，而 Gitee 上的仓库已转成只读归档，我没法跑 git 命令对比。所以这一节点到结构性差异为止，不逐条列举补丁，更不会编造"麒麟给飞腾加了 XX 调优"这种没核到的细节。

即便如此，这一节的结论已经够清楚了：麒麟内核（以 openKylin 仓库为旁证）不是主线 Linux 的简单复刻，它维护自己的 LTS 基线、加自己的 CONFIG、走自己的打包、给国产 CPU 留专门分支。这些动作拼起来，就是"自主"在内核层的具体体现——不只是跟随上游，而是有自己的维护节奏和定制层。商业版 V10/V11 的专有补丁藏在水面下（桌面版 V11 官网提到"基于全新内核开发"，具体改了什么不公开），但从开源这半边，能推断麒麟在内核层确实做了自己的事。

## 十一、管理维护理念：和其他发行版的分野

前面十章埋了不少点：国家意志的诞生、三次转身的策略、双轨格局、UKUI 的自研、合规优先的安全、政策驱动的生态、内核的自有基线。现在把它们收起来，正面回答那个核心问题——麒麟和别的发行版，到底差在哪？

先说定位。麒麟（它的开源面 openKylin 尤其强调这一点）把自己当"根社区"——不依赖别的发行版做上游，自己就是技术栈的根。Ubuntu 背后是 Debian，Fedora 自己是上游但服务于 RHEL，它们都有明确的上下游归属。麒麟要的是"我是一条根"，不是"我是某条根的下游"。这个念头决定了它维护内核基线、做自己的包格式、不肯直接用 GNOME——每一步都在强化"我是独立一极"的身份感。

治理上也能看出差别。openKylin 由开放原子开源基金会孵化，技术委员会下设 SIG 组，贡献者签 CLA，许可证是木兰宽松许可证 v2。这套东西组织色彩重——基金会托管、有架构、贡献要走协议。Debian 是全球开发者散漫自治、归 SPI 管；Fedora 背靠 Red Hat，社区和商业之间上下游分明。麒麟的治理更"有主"，不像 Debian 那样松散。

但最根本的差异在驱动力上。Ubuntu、Fedora 维护系统，图的是技术先进和社区共识：出个新功能、修个 CVE、跟上游节奏，目标是"做个好用的发行版"。麒麟维护系统，图的是自主可控和信创合规：能不能适配国产 CPU、能不能过等保、能不能进采购名录，目标是"做个合规可控的国产系统"。一个是工程逻辑，一个是制度逻辑。同一件维护内核的事，动机不同，优先级就不同，长出来的系统自然不一样。这条差异是底层的，前面三条多少都由它派生。

还有双轨这层结构。商业银河麒麟专有 + 社区 openKylin 开源，两条线并行。这和 Fedora→RHEL 的"社区上游、商业下游"是反着的——Fedora 是 RHEL 的试验田，关系线性；麒麟的商业版和社区版更像并列，各自服务不同场景。这种双轨背后，是想"既要商业落地、又要开源换生态"的平衡。

把这几点摆一起，麒麟的差异就清楚了：它不是技术水准上的差异（内核还是 Linux、包还是 deb），而是**为什么维护、为谁维护、怎么组织维护**的差异。麒麟那些显得"怪"的地方，自研桌面、合规安全、政策生态、根社区定位，都能从这套理念里推出来。理念是因，技术选择是果。

## 十二、客观评价：理念是有代价的

讲了这么多麒麟的理念，容易显得在夸。补一刀：这套理念是有代价的，而且代价不轻。

最直接的代价是生态薄。当生态靠政策和适配驱动、不靠社区自发，离开采购场景的活力就有限。一个普通开发者想给麒麟贡献个冷门应用，门槛和动力都比给 Ubuntu 贡献高。社区薄，应用就少，用户体验的天花板就低。这是理念换来的现实，没法两全。

双轨也分散精力。商业版和社区版并行，听起来两全其美，实际上两边都要维护、都要投入。openKylin 社区的活跃度，和真正的根社区（Debian、Fedora）比还有距离——"根社区"的定位和社区的实际体量之间，落差摆在那。

更隐蔽的代价是创新慢半拍。什么都握在自己手里，UKUI 自研、开明包格式自研，好处是可控，坏处是难借力。国际社区搞出来的新东西，麒麟要等自己适配、自己可控了才用得上，节奏天然慢。有时候不是做不到，是走完"可控"这道流程，黄花菜都凉了。

和同类国产系统比，麒麟和统信 UOS、Deepin 是直接竞品。Deepin 走的是"做好桌面体验、攒社区口碑"的路子，UOS 是 Deepin 的商业版。三者理念有重叠（都要自主可控），但侧重不同：麒麟偏合规和政企，Deepin 偏桌面体验和用户好感，UOS 居中。谁更好用？看场景。政企采购选麒麟的多，个人折腾桌面可能更喜欢 Deepin。

最后说句实在话：麒麟不是"最好的 Linux 发行版"，它也没打算做这个。它要的是"合规、可控、能落地"的国产系统，在它要解决的场景里，它做到了。理解麒麟，得先理解它不是为了和 Ubuntu 比好用，而是为了在特定约束下把"自主可控"做成可交付的产品。理念决定了它的样子，也划死了它的边界。

这份调查到此。一句话总结：麒麟是为国家意志和信创合规而生的 Linux，技术选择处处透着"可控优先"，这让它达成了自主可控的目标，也背着生态与活力的代价。
