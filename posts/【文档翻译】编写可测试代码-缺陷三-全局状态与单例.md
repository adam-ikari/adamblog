# 【文档翻译】编写可测试代码 · 缺陷三 · 全局状态与单例

> 译自 misko hevery 的 [Guide: Writing Testable Code](https://github.com/mhevery/guide-to-testable-code) 中的 *Flaw #3: Brittle Global State & Singletons*。为贴合 C++ 语境，代码示例由原文的 Java 改写而来，论述忠于原文。

以静态方式访问全局状态，并不能让使用全局状态的构造函数和方法的读者清楚那些共享依赖。全局状态和单例让 API 在真实依赖上说谎。要真正搞懂依赖，开发者必须逐行读代码。它会造成"超距作用"（Spooky Action at a Distance）：跑测试套件时，在一个测试里被改动的全局状态，会让后续或并行的测试出乎意料地失败。用手动或依赖注入打破这种静态依赖。

> #### 危险信号
>
> - 添加或使用单例
> - 添加或使用静态字段或静态方法
> - 添加或使用静态初始化块
> - 添加或使用注册表（registry）
> - 添加或使用服务定位器（service locator）

> **注意：**
> 本文中说"单例"或"进程级单例"时，指的是经典的设计模式（GoF）单例——也就是这种单例通过一个静态实例字段来**强制自身的"单例性"**。而"应用单例"指的是在应用中只有一个实例、但**不**强制自身"单例性"的对象。

## 为什么这是个缺陷

### 全局状态弄脏你的设计

全局状态的根本问题在于它**全局可访问**。在理想世界里，一个对象应该只能与那些被直接传给它的对象（通过构造函数或方法调用传入）交互。

换句话说，如果我实例化两个对象 A 和 B，并且从不把 A 的引用传给 B，那么 A 和 B 都没法拿到对方、也没法改动对方的状态。这是代码一种非常理想的性质。*如果我没告诉你某样东西的存在，那它就不是你能去交互的对象！* 注意全局变量和单例不是这样。对象 A 可能在开发者不知情的情况下，拿到单例 C 并改动它。如果对象 B 被实例化时也抓取了单例 C，那 A 和 B 就能通过 C 互相影响（或者说，交互会通过 C 发生重叠）。

> "使用单例的问题在于，它给系统引入了一定程度的耦合——而这种耦合几乎总是不必要的。你等于在声明，你的类只能与某一组方法的某个特定实现协作——也就是单例提供的那个实现。你不允许任何替代品。这让你的类很难脱离单例去孤立测试。测试孤立的本质，恰恰假定能为对象的合作者替换替代实现……除非你改设计，否则为了测试它的任何客户，你都被迫只能依赖单例的正确行为。"
>
> - [J.B. Rainsberger, Junit Recipes, Recipe 14.4]

### 全局状态促成"超距作用"

"超距作用"（Spooky Action at a Distance）是指：我们运行一个自以为孤立的东西（因为我们没传任何引用进去），却在我们没告诉对象的、系统远处的某些地方，发生了意料之外的交互和状态变化。这只能经由全局状态发生。举例来说，假装我们调用 `printPayrollReport(Employee employee)`。这个方法所在的类有两个字段：`Printer` 和 `PayrollDatabase`。

如果它还通过一个全局方法 `BankAccount::payEmployee(Employee, Money)` 发起银行直接存款，那非常阴险的超距作用就成为可能了。你的代码作者绝不会那么干……真的吗？

来一个没那么阴险的例子。如果 `printPayrollReport` 还顺带给 `PrinterSupplyVendor::notifyTonerUsed(int pages, double perPageCoverage)` 发了通知呢？这看上去像是个便利——打印工资单时顺便通知耗材供应商用了多少碳粉，这样你快用完前会自动收到补货。这不是便利。这是糟糕的设计，而且和所有静态物一样，它让你的代码库更难理解、演进、测试和重构。

你可能没这么想过，但每当你使用静态状态，你都在创建秘密的通信通道，而且没有在 API 里讲清楚。超距作用迫使开发者逐行读代码才能弄清潜在的交互，拉低开发效率，也把新成员搞糊涂。

不要批准使用静态、允许一个"超距"代码库的代码。优先对所需的特定合作者做依赖注入（上例中就是一个 `PrinterSupplyVendor` 实例……如果应用里只有一个，或许可以把它作为一个应用级单例来作用域管理）。

### 全局状态与单例让应用（和测试）变脆

- 静态访问妨碍与某个类的子类或包装版本协作。把依赖硬编码进去，我们就丢掉了多态的能力和灵活性。

- 每个使用全局状态的测试都需要它从一个预期状态开始，否则测试会失败。但另一个对象可能在之前的测试里已经改动了那个全局状态。

- 全局状态常常妨碍测试并行运行，这迫使测试套件跑得更慢。

- 如果你加了一个新测试（它不清理全局状态），而它跑在套件中间，那么在它之后跑的另一个测试可能会失败。

- 强制自身"单例性"的单例，最终都会作弊。
  - 你常会看到所谓的单例上有 `reset()` 或 `setForTest(...)` 这样的修改方法，因为测试中你需要更换实例。如果你测试后忘了重置单例，后面的使用就会用那个陈旧的底层实例，并且可能以一种很难调试的方式失败。

### 全局状态与单例让 API 说谎

来看一个我们想写的测试：

```cpp
// 改写自原文 Java 示例
void testActionAtADistance() {
    CreditCard card{"4444444444444441", "01/11"};
    assert(card.charge(100));
    // 但这个测试运行时会失败！
}
```

给信用卡扣款可不只是改信用卡对象的内部状态那么简单。它要求我们与外部系统对话。我们需要知道 URL，需要认证，需要在数据库里存一条记录。但这些在 `CreditCard` 的用法里一个都没讲清楚。我们说 `CreditCard` 的 API 在说谎。再试一次：

```cpp
// 改写自原文 Java 示例
void testActionAtADistanceWithInitialization() {
    // 全局状态得先初始化好
    Database::init("dbURL", "user", "password");
    CreditCardProcessor::init("http://processorurl", "security key", "vendor");

    CreditCard card{"4444444444444441", "01/11"};
    assert(card.charge(100));

    // 但这个测试还是失败！
}
```

光看 `CreditCard` 的 API，没法知道你得初始化哪些全局状态。哪怕看 `CreditCard` 的源码，也告诉不了你该调哪个初始化方法。充其量，你能找到被访问的全局变量，然后从那儿试着猜怎么初始化。

下面是修复全局状态的做法。注意初始化 `CreditCard` 的依赖变得清楚多了。

```cpp
// 改写自原文 Java 示例
void testUsingDependencyInjectedObjects() {
    Database db{"dbURL", "user", "password"};
    CreditCardProcessor processor{db, "http://processorurl",
                                  "security key", "vendor"};
    CreditCard card{processor, "4444444444444441", "01/11"};
    assert(card.charge(100));
}
```

你需要创建的每个对象，都在其构造函数的 API 里声明了依赖。要构建测试 `CreditCard` 所需的对象，已经不再含糊。

通过显式声明依赖，哪些对象需要实例化就清楚了（这里就是 `CreditCard` 需要一个 `CreditCardProcessor`，而它又需要一个 `Database`）。

### 全局性与"全局负载"是传递的

我们可以把*"全局负载"*定义为：通过全局状态暴露出来、可被（直接或间接）改动的变量有多少。数量越大，问题越大。下面这个全局负载是 1。

```cpp
// 改写自原文 Java 示例
class UniqueID {
public:
    // 全局负载为 1，因为只有 nextID 暴露为全局状态
    [[nodiscard]] static int get() {
        return nextID_++;
    }

private:
    static int nextID_;  // 定义于 .cpp：int UniqueID::nextID_ = 0;
};
```

下面这个例子的全局负载会是多少？

```cpp
// 改写自原文 Java 示例
class AppSettings {
public:
    static AppSettings& instance() {
        static AppSettings inst;  // Meyers 单例
        return inst;
    }

    int numberOfThreads = 10;
    int maxLatency = 20;
    int timeout = 30;

private:
    AppSettings() = default;  // 阻止外部实例化
};
```

这里的问题更复杂一点。实例字段通过一个静态方法暴露。遍历这个全局实例，我们暴露了三个变量：`numberOfThreads`、`maxLatency` 和 `timeout`。一旦我们访问一个全局变量，所有通过它访问到的变量也就都变成全局的了。全局负载是 3。

从行为角度看，一个直接通过 static 声明的全局变量，和一个经传递变全局的变量，没有区别。它们同样有害。一个应用可能只有很少几个静态变量，却能经传递累积出很高的全局负载。

### 单例是披着羊皮的全局状态

多数软件工程师都同意全局状态是不理想的。然而，单例制造全局状态，却仍有那么多人在新代码里用。对抗这种趋势，鼓励人们用别的机制替代经典的进程级单例。我们常常并不真的需要单例（如今对象创建相当便宜）。如果你需要保证每个应用只有一个共享实例，用依赖注入容器的单例作用域。

### "但我的应用只有一个单例"是无意义的

下面是一个典型的 `Cache` 单例实现。

```cpp
// 改写自原文 Java 示例
class Cache {
public:
    static Cache& instance() {
        static Cache inst;
        return inst;
    }

    std::unordered_map<std::string, User> userCache;
    EvictionStrategy* eviction = new LruEvictionStrategy();  // 示例：内部状态

private:
    Cache() = default;
};
```

这个单例的全局负载高得没有上限（我们能在 `userCache` 里放无限多个 `user`）。`Cache` 单例把 `std::unordered_map<std::string, User>` 暴露进了全局状态，于是每个 user 都成了全局可见。此外，每个 `User` 的内部状态、还有 `EvictionStrategy` 也作为全局可变状态暴露出来。可以看到，一个无辜的对象多么容易就跟全局状态纠缠在一起。**"但我的应用只有一个单例"这类说法是无意义的，因为*暴露出的全局状态总量*是从显式全局状态可达的对象的传递闭包。**

### 应用运行时里的全局状态可能"感觉还好"

生产环境我们实例化应用的一个实例，所以从状态冲突的角度看，全局状态不是问题（弄脏设计的问题仍然存在——见上文）。在一个进程里实例化同一应用的两份副本并不常见，所以生产里有全局状态可能"感觉还好"。（一个例外是 web 服务器。在不同端口上跑多个 web 服务器实例很常见。全局状态在那儿可能会有问题。）下面我们会看到，这种"还好"的感觉是误导人的，它其实是一头阴险的野兽。

### 测试运行时里的全局状态是一头阴险的野兽

测试时，每个测试都是应用的一次孤立、部分实例化。没有外部状态进入测试（没有外部对象传进测试的构造函数或测试方法）。也没有状态离开测试（测试方法是 void，什么都不返回）。一个理想的测试完成时，与该测试相关的所有状态都消失。这让测试孤立，它创建的所有对象都可被回收。此外，创建出的状态局限于当前线程。这意味着我们能并行或以任意顺序运行测试。然而，当存在全局状态/单例时，所有这些美好的假设都崩塌。状态能进出测试，而且不被回收。这让测试的顺序变得要紧。你没法并行跑测试，测试还会因为线程交互变得 flaky。

真正的单例很可能根本没法测。结果是，多数试图测试带单例的应用的开发者，会用两种方式放宽他们单例的"单例性"。（1）从实例的静态 final 声明里去掉 `final` 关键字。这让他们能为不同测试替换不同的单例。（2）他们提供第二个 `initializeForTest()` 方法，允许修改单例状态。然而，这些方案充其量只是个*应急手段*，产出的是难维护、难理解的代码。每个影响全局状态的测试（或 tearDown）都必须撤销这些改动，否则就泄漏给后续测试。而且如果并行跑测试，测试孤立几乎不可能。

全局状态是单元测试单一最大的头疼来源！

## 识别这个缺陷

- 症状：存在静态字段
- 症状：CL 里的代码做静态方法调用
- 症状：一个单例有 `initializeForTest(...)`、`uninitialize(...)` 等重置方法（即告诉它用某个轻量级服务，而不是那个去连别的服务器的）
- 症状：测试在套件里跑会失败，但单独跑通过，反之亦然
- 症状：改变执行顺序，测试就失败
- 症状：读取或写入 flag 值，或调用了 `Flags::disableStateCheckingForTest()` 和 `Flags::enableStateCheckingForTest()`
- 症状：CL 里的代码有或使用了 Singletons、Mingletons、Hingletons、Fingletons（见 [Google Singleton Detector](https://web.archive.org/web/20200514140130/http://code.google.com/p/google-singleton-detector/wiki/WhySingletonsAreControversial)）

**"进程级全局状态"里的"全局"和"应用共享状态"里的"全局"是两回事。**

- 进程级全局状态出现在用 `static` 关键字使一个字段或方法可访问、并返回一个共享对象时。用 static 来促成共享状态，就是问题所在。因为 static 被强制为"一个进程一个"，并行化和隔离测试就成了大问题。从维护角度看，静态字段制造耦合、隐藏的合作者，以及谎报真实依赖的 API。*静态访问是问题的根源。*

- 应用共享状态仅仅意味着同一个实例在代码的多个地方被共享。在任意时刻一个类可能有也可能没有多个实例，取决于应用逻辑是否强制唯一性。共享状态并不通过 `static` 全局访问。它被传给需要该状态的合作者，或由依赖注入容器管理所需共享状态的一致注入（即：在处理一个请求期间，同一个 `User` 需要被注入到服务于该用户请求的线程里的多个对象中。容器会把它作用域化为请求级）。有问题的不是共享状态本身。应用里有些地方需要访问共享状态。通过*静态物*来共享那种状态，才会让代码变脆、让测试变难。

*进程级全局状态的检验：*

回答下面这个问题："理论上，我能在同一个进程里创建你应用的第二个实例，而且不发生任何冲突吗？" 如果不能，那你就在用进程级全局状态（用了 `static` 关键字，让进程强制单例的单例性）。共享状态请改用依赖注入。

## 修复这个缺陷

> 依赖注入是你的朋友。
> 依赖注入是你的朋友。
> 依赖注入是你的朋友。

- 如果你需要一个合作者，用依赖注入（把合作者传进构造函数）。依赖注入会让你的合作者清晰，并给你注入测试替身的接缝。

- 如果你需要共享状态，用依赖注入容器，它能在完全可测的前提下管理应用作用域单例。

- 如果某个静态物在代码库里被广泛使用，而你没法在一个 CL 里把它替换成容器管理的单例作用域对象，试试下面这些权宜之计之一：

  - 创建一个适配器（Adapter）类。它大概只会有一个默认构造函数，适配器的方法每个都和你要解耦的静态方法同名（并调用转发到那个静态方法）。这并没完全修好问题——静态访问仍然存在，但至少适配器在测试中可以被伪造/打桩。一旦某个静态方法或装满静态方法的工具类的所有消费者都被适配了，就能通过把共享行为/状态推进适配器（把它们从适配器变成真正的合作者）来消除静态物。用应用逻辑或容器作用域来强制必要的共享。

  - 与其给静态方法包一层适配器，有时你可以更早地把共享行为/状态挪进一个可实例化的类。让容器管理这个可实例化对象（或许在单例作用域里），并把一个容器管理的实例放在那个静态方法背后，直到所有调用方都能被重构为注入实例、而非通过静态方法使用它。同样，这是个仍保留静态物的半拉子方案，但它是朝向移除静态物迈出的一步，在处理铺天盖地的静态方法时可能有用。

  - 用小步消除单例时，试着把一个 Provider 绑定到你想在单例作用域里共享的类上（或用一个标注了单例的 provider 方法）。Provider 返回它从 GoF 单例取到的实例。尽可能用容器注入这个共享实例，一旦所有注入点都能注入，就消掉那个 GoF 单例。

  - 如果你被一个库类的静态方法困住了，把它包进一个实现了接口的对象里。把这个对象传到需要它的地方。你可以在测试中给接口打桩，从而切掉静态依赖。见下例。

  - 如果用依赖注入容器，你可以把 flag 值绑定到可注入对象上。然后凡是你需要这个 flag 的值的地方，注入它。对测试，你可以用依赖注入传入任意值，完全绕开 flag，从而轻松并行化。

// 改写自原文 Java 示例

class LoginService {
public:
    static LoginService& getInstance() {
        if (instance_ == nullptr) {
            instance_ = std::make_unique<RealLoginService>();
        }
        return *instance_;
    }

    // 在测试开头调这个
    static void setForTest(std::unique_ptr<LoginService> testDouble) {
        instance_ = std::move(testDouble);
    }

    // 在测试结尾调这个，否则你就有风险把 testDouble
    // 留作后续测试的单例。
    static void resetForTest() {
        instance_.reset();
    }

    virtual ~LoginService() = default;
    virtual bool isAuthenticatedAdmin(const User& user) = 0;
    // ...

protected:
    LoginService() = default;

private:
    static std::unique_ptr<LoginService> instance_;  // 定义于 .cpp，初值 nullptr

class AdminDashboard {
public:

};

出于各种原因，你的应用里可能确实只需要某样东西的一个实例。这通常实现为单例 [GoF]，类能给出一个对象实例，且不可能同时实例化两个。这样一个进程级单例是有代价的，代价就是灵活性和可测试性。人们可能用 `setForTest(...)` 和 `resetForTest()` 方法（通过破坏封装）去改动底层单例的实例，从而绕开这些问题。

- 缺陷：和所有用 `static` 方法的地方一样，没有接缝去多态地改变实现。你的代码变得更脆。
- 缺陷：测试没法并行跑，因为每个线程对共享全局状态的改动都会冲突。
- 缺陷：`@VisibleForTesting` 是个信号，提示这个类该被重构，使之不需要为了测试而破坏封装。注意解决方案里它被移除了。

如果你需要"只有一个实例"的保证，告诉依赖注入容器那个对象在单例作用域里。容器管理的单例不是设计问题，因为测试里你可以创建多个实例（并行跑、避免交互、用不同配置）。在生产运行时，容器会确保注入的是同一个实例。

> 又名：*调用 `Flags::disableStateCheckingForTest()` 和 `Flags::enableStateCheckingForTest()` 时闻到的那股馊味。*

带静态字段的 flag 类，被当作一种共享在应用启动时确定的设置（以及共享全局状态）的方式。但和所有全局状态一样，它们代价沉重。flag 有一个严重缺陷。因为它们共享全局状态，测试前后需要非常小心地调整（否则后续测试可能失败）。

- 缺陷：一个测试可以设一个 flag 值然后忘了重置，导致后续测试失败。
- 缺陷：如果两个测试需要某个 flag 的不同值才能跑，你不能并行化它们。强行并行的话，哪个线程设置 flag 值会有竞态，另一个线程的测试就会失败。
- 缺陷：需要 flag 的代码很脆，而且它的消费者看 API 不知道到底用没用 flag。API 在对你撒谎。

要治这些病，求助于我们的朋友依赖注入。你可以用容器去发现并使任意给定类里的所有 flag 可注入。然后你就能自动注入所需的 flag *值*，而从不引用那些静态 flag 变量。因为你操作的是普通对象（不是 flag），就不再需要调 `Flags::disableStateCheckingForTest()` 或 `Flags::enableStateCheckingForTest()`。

有这类问题的类，测试单独跑可能通过，但在套件里跑就失败。测试在套件里改了顺序，也可能失败。

取决于测试执行顺序，谁先触发 `RpcClient` 加载，谁就会让 `flag_useRealBackend` 被读取、值被永久定下。后面想用别的 backend 的测试就不行了，因为静态物让全局状态能在测试的 setup 和 teardown 之间存留。

如果你通过给 `RpcClient` 的 `Backend` 暴露一个 setter 来绕开，你会撞上和上面"问题：需要调用 `setForTest(...)` 和/或 `resetForTest()` 方法"一样的问题。静态物的根本问题没解决。

- 缺陷：静态初始化块只跑一次，且测试无法覆盖。
- 缺陷：`Backend` 被设一次，再也不可为后续测试更改。这可能导致某些测试失败，取决于测试顺序。

要治这些病，先移除静态状态。然后把 `RpcClient` 需要的 `Backend` 注入进去。依赖注入再次救场。用容器在应用作用域里管理 `RpcClient` 的那个唯一实例。摆脱进程级单例让测试全方位变容易。

有时你会被一个库里的静态方法困住，需要在测试里阻止它跑。但你又需要这个库，所以没法移除或替换成非静态实现。因为它是库，你没有控制权去移除 static 修饰符、把它变成实例方法。

- 缺陷：即使你不想，你也被迫执行 `TrackStatusChecker` 的方法，因为一个静态调用把它锁死在那儿了。
- 缺陷：测试可能更慢，而且冒着通过库里的静态物改动全局状态的风险。
- 缺陷：静态方法不可覆盖、不可注入。
- 缺陷：静态方法从你的测试代码里移除了一个接缝。

如果你能控制代码（它不是第三方库），你要修根上的问题，移除静态方法。

如果你没法改外部库，把它包进一个实现了相同接口的类里（或建你自己的接口）。然后你就能为这个包装好的库注入一个测试替身，它有特定于测试的不同行为。这是更好的设计，也更容易测。我们常常发现，可测的代码就是质量更高、更易维护、更高效的代码。还要考虑，如果你建自己的接口，你可能不需要支持一个库类里的每个方法——只适配你实际用到的功能。

## 告诫：全局状态什么时候 OK？

有两种情况全局状态是可容忍的。

1. 当*引用是个常量*，且它指向的对象要么是*基本类型*要么*不可变*。所以 `static const std::string URL = "http://google.com";` 是 OK 的，因为没法改这个值。记住，你指向的所有对象的传递闭包也必须不可变，因为全局性是传递的。`std::string` 是安全的，但把它换成一个 `MyObject` 就有风险了，因为 `MyObject` 暴露的所有状态的传递闭包可能可变。如果将来有人在你的不可变对象上加可变状态，那你无辜的代码就变成头疼事，你就站在薄冰上了。

2. 当*信息只单向流动*时。比如一个 `Logger` 就是个大单例。但我们的应用只往 logger 里写、从不读它。更重要的是，我们的应用不会因为 logger 里启用了什么、没启用什么而表现不同。虽然从测试角度看这不是问题，但如果你想去断言你的应用确实记下了重要消息，那就是个问题了。因为测试没法把 `Logger` 换成一个测试替身（我知道我们能给自己的日志级别设 handler，但那些只不过是上面展示的更多同类问题）。如果你要测 logger，就把类改成依赖注入 `Logger`，这样你就能插入一个 `MockLogger` 并断言正确的东西被写进了 `Logger`。（作为便利，依赖注入容器自动知道怎么为任意类构造注入一个配置好的 logger，只要把它加到构造函数参数里，正确的那个就会被传进来。）
